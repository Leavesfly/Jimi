# 10 · 记忆管理与会话机制

> 本篇讲清楚 Jimi 如何在**跨会话**层面"记住"你的项目、如何把**单次会话**持久化到磁盘、以及上下文 token 超限时是怎样**自动压缩**的。所有结论基于源码 `src/main/java/io/leavesfly/jimi/memory/**`、`src/main/java/io/leavesfly/jimi/core/session/**`、`src/main/java/io/leavesfly/jimi/core/compaction/**`、`src/main/java/io/leavesfly/jimi/core/engine/context/ContextManager.java` 亲读确认，未出现的字段/命令一律视为"未实现"。

---

## 1 · 三大子系统地图

| 子系统 | 粒度 | 生命周期 | 落盘位置 | 主要入口 |
|---|---|---|---|---|
| **Memory（长期记忆）** | 跨会话、按**工作目录**分片 | 持久（除非手动清空） | `~/.jimi/memory/{hex(workDir.hashCode())}/MEMORY.md` + `topics/*.md` | `MemoryManager` + `/memory` 命令 + `MemoryTool` |
| **Session（单次会话）** | 单次 REPL 从启动到 `/new` / `/quit` | 持久（JSONL） | `~/.jimi/sessions/{hex(workDir.hashCode())}/{UUID}.jsonl` | `SessionManager` + `/new` 命令 |
| **Context（上下文窗口）** | Agent 运行时的 token 窗口 | 内存（快照由 Context 自行管理检查点） | 写入 session 的 `.jsonl` 文件 | `ContextManager#checkAndCompact` + `Compaction` 接口 |

三者的联动关系是：Session 提供一个"历史文件句柄"给 Context（`new Context(historyFile, ObjectMapper)`），Context 每次 `appendMessage` 都会持久化为 JSONL；Memory 独立于 Context，在 System Prompt 层面**注入**长期记忆文本。

## 2 · Memory（长期记忆）—— 三层架构

`MemoryManager` 的 JavaDoc 明确对标 Claude Code 的三层记忆模型：

| 层 | 文件 | 加载时机 | 用途 |
|---|---|---|---|
| **Layer 1** | `MEMORY.md` | **始终**注入 System Prompt | 索引层，记录项目整体偏好/决策/教训 |
| **Layer 2** | `topics/*.md` | **按需**加载（通过 `TopicMatcher` 匹配用户输入） | 主题文件层，细粒度知识 |
| **Layer 3** | `.jsonl` 会话记录 | **仅 grep 搜索** | 历史会话全文检索（`MemorySearcher`） |

### 2.1 存储路径计算

`MemoryStore` 构造函数（`memory/MemoryStore.java#43-52`）：

```java
public MemoryStore(String workDirPath, String customStoragePath) {
    if (customStoragePath != null && !customStoragePath.isEmpty()) {
        this.memoryRoot = Paths.get(customStoragePath);
    } else {
        String dirHash = Integer.toHexString(workDirPath.hashCode());
        this.memoryRoot = Paths.get(System.getProperty("user.home"), ".jimi", MEMORY_DIR, dirHash);
    }
    ensureDirectories();
}
```

**关键事实**：
- **路径隔离**用的是 `Integer.toHexString(workDirPath.hashCode())`——**不是加密 hash**。Java 的 `String.hashCode()` 是 32 位 int，理论上存在碰撞风险；但对"一个用户的几十个工程目录"规模，碰撞概率极低。
- `ensureDirectories` 会在构造器里主动 `Files.createDirectories(memoryRoot)` 和 `memoryRoot/topics/`。
- `MemoryManager` 对 `MemoryStore` 做了**按 workDirPath 的 ConcurrentHashMap 缓存**（`storeCache`），同一工作目录多次访问复用 store 实例。

### 2.2 `MEMORY.md` 的默认模板

首次进入一个工作目录时，`MemoryStore.initializeIfAbsent(workDirPath)` 会写入：

```markdown
# Project Memory

## Project Overview
- **Project**: {项目目录名}
- **Initialized**: {ISO_LOCAL_DATE}

## User Preferences

## Key Decisions

## Lessons Learned
```

之后 `MemoryManager.getMemorySummary(workDirPath)` 会把该 MD 内容包裹到：

```
## Long-term Memory

The following is your persistent memory about this project and user preferences. Use this information to provide more contextual and personalized assistance.

{MEMORY.md 原文}
```

**注入时机**：在 System Prompt 构建路径上（见 02 篇 §3.2 的 System Prompt 组装）。

### 2.3 写记忆的三条路径

源码 `MemoryManager` 暴露三组写入 API：

| API | 对 section 的处理 | 使用者 |
|---|---|---|
| `writeMemory(workDir, section, content)` | 若 section 存在 → 替换到下一个 `## ` 前；不存在 → 追加到末尾 | `/memory write`、`MemoryTool#write` |
| `appendMemory(workDir, section, entry)` | 若 section 存在 → 在下一个 `## ` 前插入 `- entry`；不存在 → 创建并追加 | `MemoryExtractor` 每轮调用 |
| `overwriteMemory(workDir, content)` | 完全覆盖 MEMORY.md | `/memory clear`、`MemoryConsolidator` 整理后 |

`updateSection` 和 `appendToSection` 都用 `indexOf("\n## ")` 定位下一个 section，**不支持深层（###）标题**——MEMORY.md 的结构必须保持 `## ` 顶层划分。

### 2.4 `MemoryExtractor`（Per-turn capture）

每轮 Agent 执行**成功**后由 `AgentExecutor.onExecutionSuccess()` 调用（JavaDoc 明示）。核心流程：

```java
if (!config.isAutoExtract()) return;   // 开关
// 1. 读 ExecutionState.keyDecisions → 每条用 isSignificant 筛选（entry != null && entry.trim().length() > 10）
// 2. 读 ExecutionState.lessonsLearned → 同上 isSignificant 筛选
// 3. 读 ExecutionState.filesModified → 当 files.size() >= 2 才汇总成一条 "Modified N files: ..."
//                                      （只改 1 个文件的执行直接跳过，不提取）
// 4. 每条前缀 ISO_LOCAL_DATE 日期（如 "2026-05-03: <entry>"）
// 5. 按关键词路由到 section（detectSection）
for (String entry : extractedEntries) {
    memoryManager.appendMemory(workDirPath, detectSection(entry), timestampedEntry);
}
```

其中 `isSignificant` 的完整源码：

```java
private boolean isSignificant(String entry) {
    return entry != null && entry.trim().length() > 10;
}
```

——trim 后长度 ≤ 10 字符的条目被丢弃，即 **"短决策"/"短教训" 不会被记录**。

**`detectSection` 的路由规则（关键词匹配）**：

| 关键词命中（lowercase contains） | 路由到 |
|---|---|
| `决策` / `decision` / `选择` / `chose` | `Key Decisions` |
| `教训` / `lesson` / `注意` / `avoid` | `Lessons Learned` |
| `modified` / `修改` | `Key Decisions`（**注意**：这里不是 `File Changes` 或 `Lessons Learned`） |
| 其他 | `Key Decisions`（**兜底**） |

**观察**：所有无法识别的条目都落到 `Key Decisions`——`User Preferences` 和 `Project Overview` 这两个 section 实际上**永远不会被 `MemoryExtractor` 自动写入**，只能通过 `/memory write` 手动维护。

### 2.5 `MemoryConsolidator`（autoDream）

对标 Claude Code 的 **autoDream** 后台整理机制。触发条件（`shouldConsolidate`）：

```java
if (sessionCount < config.getConsolidateMinSessions()) return false;   // 默认 5
Instant last = getLastConsolidationTime(workDirPath);
if (last != null && hoursSince(last) < config.getConsolidateMinHours()) return false;   // 默认 24
return true;
```

即**会话数 ≥ 5 且距离上次整理 ≥ 24 小时**才整理。时间戳存在 `memoryRoot/.last_consolidation` 文件。

**Phase 编号（源码注释写明）**：
1. **Orient**：读 MEMORY.md 现状。
2. **Gather signal**：从 MEMORY.md 里 grep "带日期前缀的条目"（`- YYYY-MM-DD: ...`）作为"近期信号"。
3. **Consolidate**：
   - 有 LLM → 拼 `CONSOLIDATION_PROMPT` 调 `llm.getChatProvider().generate(...)`，要求输出以 `# Project Memory` 开头的整理后全文，**限 2000 tokens**。
   - 无 LLM → **规则整理**：保留所有 `#` 标题和非 `- ` 开头行；对 `- ` 条目保留前 `maxMemoryTokens / 10` 条。
4. **Prune & index**：`overwriteMemory` 覆盖 + 更新 `.last_consolidation`。

**并发保护**：`ConcurrentMap<String, AtomicBoolean> consolidationLocks` 按 workDirPath 加锁，`compareAndSet(false, true)` 失败直接返回 `Mono.empty()`。

**关键校验**：LLM 回复必须 `contains("# Project Memory")` 才写回，否则 `log.warn` 跳过——**防止 LLM 返回无关内容污染记忆**。

### 2.6 `/memory` 命令分派

`MemoryCommandHandler.execute` 子命令路由：

| 子命令 | 行为 |
|---|---|
| （空） | `showMemorySummary`——前 500 字符 + Topic 列表 |
| `read` | 打印完整 MEMORY.md |
| `topics` | 列出 `topics/` 下所有 `.md` 文件 |
| `search <q>` | `new MemorySearcher().searchAll(q, 5)` 搜全部会话 `.jsonl` |
| `write <section> <content>` | `memoryManager.writeMemory`（`buildQueryFromArgs` 从 index 2 起用空格拼接所有剩余 arg，**但注意**：`MetaCommandProcessor` 层面用 `split("\\s+")` 切分，引号不保留——见 09 篇 §3.1 "参数解析两处典型坑"。这意味着 `/memory write "User Preferences" 内容` 实际被切成 `[write, "User, Preferences", 内容]`，section 会拿到 `"User` 这个含引号的破碎值） |
| `clear` | `memoryManager.overwriteMemory(workDir, "")` 清空（**不是删文件**，只是内容置空） |
| `help` | `showHelp(out)` 打印 6 行帮助文本（包含 `/memory help` 自己） |
| 其他 | `printError("未知子命令: " + subCommand)` + `showHelp(out)` |

**注意**：源码 `getUsage()` 返回 `/memory [read|topics|search <query>|write <section> <content>|clear]`——**usage 字符串里没列出 `help`**，但源码 `switch` 分支真实支持它。这是**帮助文本与 usage 字符串之间的不一致**（类似 09 篇 H1 的文案漏收），见 §7.2 M-S6。

**禁用开关**：`memoryManager.getConfig().isEnabled() == false` 时直接打印 `"记忆系统未启用。请在 config.yml 中设置 memory.enabled: true"` 并返回。

### 2.7 `MemoryConfig` 字段清单

`config/info/MemoryConfig.java` 全量字段：

| 字段 | 默认 | 说明 |
|---|---|---|
| `enabled` | `true` | 主开关 |
| `storagePath` | `""` | 空串 → 使用 `~/.jimi/memory/{hash}`，非空 → 使用指定绝对路径（此时**所有 workDir 共享一个 store** —— 相当于全局记忆） |
| `maxMemoryTokens` | `2000` | 整理时的目标 token 数上限（LLM prompt 提及，规则整理也用 `/10` 约束条目数） |
| `autoExtract` | `true` | Per-turn 自动提取开关 |
| `autoConsolidate` | `true` | autoDream 自动整理开关 |
| `consolidateMinSessions` | `5` | 触发整理的最少会话数 |
| `consolidateMinHours` | `24` | 触发整理的最少间隔小时 |

## 3 · Session（会话持久化）

### 3.1 元数据集中存储

`SessionManager` 的元数据容器 `Metadata` 序列化到 `~/.jimi/metadata.json`，结构：

```json
{
  "work_dirs": [
    {
      "path": "/abs/path/to/project",
      "last_session_id": "uuid-1",
      "session_ids": ["uuid-1", "uuid-0", ...]
    },
    ...
  ]
}
```

`@JsonProperty("work_dirs")` 指定 JSON key（snake_case），Java 字段用 camelCase。`saveMetadata` 用 `writerWithDefaultPrettyPrinter()` 美化输出。

### 3.2 会话文件路径

`WorkDirMetadata#getSessionsDir`：

```java
public Path getSessionsDir() {
    String dirHash = Integer.toHexString(path.hashCode());
    return Paths.get(System.getProperty("user.home"), ".jimi", "sessions", dirHash);
}
```

会话文件路径 = `getSessionsDir() / "{UUID}.jsonl"`。注意：**这里用的是 `path`（即 workDir 的字符串路径）而不是 `memoryRoot`**——Memory 和 Session **共用 `Integer.toHexString(workDirPath.hashCode())` 的哈希规则**，但子目录不同（`memory/` vs `sessions/`），**哈希值本身是完全一致的**。

### 3.3 `createSession` 流程

```java
public Session createSession(Path workDir) {
    Metadata metadata = loadMetadata();                           // 读 metadata.json
    WorkDirMetadata workDirMeta = findOrCreateWorkDirMetadata(...);// 找或新建 workDir 项
    String sessionId = UUID.randomUUID().toString();
    Path historyFile = workDirMeta.getSessionsDir().resolve(sessionId + ".jsonl");
    Files.createDirectories(historyFile.getParent());
    if (Files.exists(historyFile)) {
        log.warn("History file already exists, truncating: {}", historyFile);
        Files.delete(historyFile);
    }
    Files.createFile(historyFile);
    workDirMeta.setLastSessionId(sessionId);                      // 更新 last_session_id
    workDirMeta.getSessionIds().add(sessionId);                   // 追加到 session_ids
    saveMetadata(metadata);                                        // 写回 metadata.json
    loadAgentsMd(workDir);                                         // 预加载 AGENTS.md 到缓存
    return Session.builder().id(sessionId).workDir(workDir.toAbsolutePath()).historyFile(historyFile).build();
}
```

**观察**：
- **每次都完整重写 metadata.json**，并发多实例启动时存在丢失风险（无锁）。Jimi 的场景是单进程 REPL，这个风险可以忽略。
- UUID 撞车时会**删旧文件再建新**（`log.warn` 但不抛异常）——这个分支理论上永远不会命中（UUID v4 碰撞概率 ≈ 0）。

### 3.4 `continueSession` 流程

```java
public Optional<Session> continueSession(Path workDir) {
    Metadata metadata = loadMetadata();
    Optional<WorkDirMetadata> workDirMeta = findWorkDirMetadata(metadata, workDir);
    if (workDirMeta.isEmpty()) return Optional.empty();
    String lastSessionId = workDirMeta.get().getLastSessionId();
    if (lastSessionId == null) return Optional.empty();
    Path historyFile = workDirMeta.get().getSessionsDir().resolve(lastSessionId + ".jsonl");
    return Optional.of(Session.builder().id(lastSessionId).workDir(...).historyFile(historyFile).build());
}
```

**重点**：`continueSession` **不检查历史文件是否真实存在**——如果用户手工删除了 `.jsonl` 文件但没清 metadata，仍会返回一个指向"幽灵文件"的 Session；Context 读该文件时会拿到空历史（按 `Context` 的容错逻辑）。

### 3.5 `Session` 实体字段

| 字段 | 类型 | 作用 |
|---|---|---|
| `id` | String (UUID) | 会话标识 |
| `workDir` | Path (absolute) | 工作目录 |
| `historyFile` | Path | `.jsonl` 文件 |
| `createdAt` | Instant (默认 `now()`) | 创建时间（仅记录，未被任何命令展示） |
| `globalStepCounter` | AtomicInteger | 跨 Agent 共享的全局步数计数 |
| `cancelled` | AtomicBoolean | 取消标志（Ctrl-C 处理） |

`incrementAndGetGlobalStep` / `cancel` / `resetCancelled` 这些方法在 REPL 的 step 计数和中断流程中被使用。

### 3.6 `AGENTS.md` 的加载与缓存

`SessionManager#loadAgentsMd` 在创建会话时预加载：

```java
Path agentsPath = workDir.resolve("AGENTS.md");
Path agentsPathLower = workDir.resolve("agents.md");
// 优先大写，其次小写
```

缓存在 `Map<String, String> agentsMdCache`（按 workDir 绝对路径），`clearAgentsMdCache()` 全清，`clearAgentsMdCache(workDir)` 按目录清。这个缓存被 System Prompt 组装路径使用（03 篇 §2 有提及）。

### 3.7 `/new` 命令

`NewCommandHandler.execute`：

```java
String oldSessionId = context.getEngineClient().getSessionId();
context.getEngineClient().newSession().block();                   // 创建新会话
String newSessionId = context.getEngineClient().getSessionId();
out.printSuccess("✅ 新会话已创建");
out.printInfo("  旧会话: " + oldSessionId);
out.printInfo("  新会话: " + newSessionId);
```

`WireEngineClient#newSession` 的内部实现：

```java
Session newSession = sessionManager.createSession(workDir);       // 创建并切换
this.sessionId = newSession.getId();
return wire.request(new SessionResetRequest())                    // 通知 Engine 重置上下文
        .doOnSuccess(v -> { /* 重置 Wire Sink */ });
```

**关键**：`/new` 不只是清空上下文，还会**真正创建一个新的 `.jsonl` 文件**——旧会话的 `.jsonl` 保留在磁盘上，未来可通过 `/memory search` 检索。

### 3.8 `/reset` 命令

`ResetCommandHandler.execute`：

```java
int checkpoints = context.getEngineClient().getContextInfo().getCheckpointCount();
if (checkpoints == 0) {
    out.printInfo("上下文已经为空");
    return;
}
context.getEngineClient().resetContext().block();
out.printSuccess("✅ 上下文已清除");
out.printInfo("已回退到初始状态，所有历史消息已清空");
```

`WireEngineClient#resetContext` 的实现**只是**：

```java
return wire.request(new SessionResetRequest());
```

**`SessionResetRequest` 的处理方**是 `WireRequestHandler`，它会让当前 Context 回退到最初检查点——**但不创建新 `.jsonl` 文件**，历史消息被**写入同一个 `.jsonl`**之后被 Context 逻辑丢弃（Context 通过 `revertTo(0)` 只保留首检查点的消息）。

**`/reset` 与 `/new` 的区别**：

| 维度 | `/reset` | `/new` |
|---|---|---|
| 创建新 `.jsonl` | ❌ 否 | ✅ 是 |
| 更新 metadata.json | ❌ 否 | ✅ 是（新的 `last_session_id`） |
| 上下文回退 | ✅ 回到初始检查点 | ✅ 完全空上下文 |
| 回到 REPL 后重启 `--continue` 会看到什么 | 被清空的历史（因为写到同一文件并被 `revertTo(0)` 覆盖） | 全新的空会话 |

### 3.9 `/history` 命令

`HistoryCommandHandler` 读的**不是 `.jsonl`**，而是 **JLine 的 `LineReader.getHistory()`**——即当前 REPL 进程内用户敲过的**命令行输入**：

```java
for (History.Entry entry : context.getLineReader().getHistory()) {
    out.println(String.format("  %3d  %s", index++, entry.line()));
}
```

**观察**：JLine 的 history 默认存在 `~/.jline_history` 或 `~/.jimi/.history`（由 JLine 配置决定），和 Session 的 `.jsonl` **完全是两套东西**——前者是用户的按键历史，后者是 Agent 的对话历史。

### 3.10 `/status` 命令

```java
out.println("  Agent: " + context.getEngineClient().getAgentName());
out.println("  可用工具数: " + context.getEngineClient().getToolNames().size());
ContextQueryRequest.ContextInfo contextInfo = context.getEngineClient().getContextInfo();
out.println("  上下文消息数: " + contextInfo.getHistorySize());
out.println("  上下文 Token 数: " + contextInfo.getTokenCount());
```

数据源是 `ContextQueryRequest.ContextInfo`（内嵌类，字段 `tokenCount` / `historySize` / `checkpointCount`），通过 `wire.request(new ContextQueryRequest())` 发给 Engine，由 `WireRequestHandler#handleContextQuery` 从 `Context` 读取三项计数。

**`/status` 不显示 `checkpointCount`**——虽然接口返回了，但 handler 没打印。

## 4 · Context（上下文压缩）

### 4.1 `Compaction` 接口与 `SimpleCompaction` 唯一实现

接口极简（`core/compaction/Compaction.java`）：

```java
public interface Compaction {
    Mono<List<Message>> compact(List<Message> messages, LLM llm);
}
```

仓库中只有一个实现 `SimpleCompaction`（`@Service`），即"简单压缩"策略。

### 4.2 `SimpleCompaction` 的保留策略

源码 `SimpleCompaction#compact` 逻辑：

```java
private static final int MAX_PRESERVED_MESSAGES = 2;   // 保留最近 2 条 USER/ASSISTANT
// 1. 从末尾往前数 USER/ASSISTANT 消息，数到第 2 条的位置 preserveStartIndex
// 2. 如果 USER/ASSISTANT 不足 2 条 → 直接返回原 messages（不压缩）
// 3. toCompact = messages[0, preserveStartIndex)
//    toPreserve = messages[preserveStartIndex, end)
// 4. 把 toCompact 序列化为 Markdown 文本（## Message N / Role: ... / Content: ...）
// 5. 调 llm.getChatProvider().generate(COMPACT_TEMPLATE, ...)，获得压缩后文本
// 6. 新结果 = [1 条 ASSISTANT 消息（内含 "Previous context has been compacted..." + 压缩文本）] + toPreserve
// 7. onErrorResume → 返回原 messages（压缩失败不丢消息）
```

**硬编码项**：
- `MAX_PRESERVED_MESSAGES = 2`（保留最后 2 条 USER/ASSISTANT）
- `COMPACT_TEMPLATE`（中文 prompt，固定在源码）
- 压缩结果以 **ASSISTANT** 角色写回（**不是** SYSTEM，因为部分 LLM 不接受多 SYSTEM）

### 4.3 真正触发压缩的路径

**`/compact` 命令不触发真实压缩**（09 篇 §8 L1 已说明）。真正的压缩触发点在 `ContextManager#checkAndCompact`：

```java
public Mono<Void> checkAndCompact(Context context, LLM llm, Compaction compaction) {
    if (llm == null || compaction == null) return Mono.empty();
    int currentTokens = context.getTokenCount();
    int maxContextSize = llm.getMaxContextSize();
    if (currentTokens > maxContextSize - EngineConstants.RESERVED_TOKENS) {
        wire.send(new CompactionBegin());
        return compaction.compact(context.getHistory(), llm)
                .flatMap(compactedMessages -> context.revertTo(0)
                        .then(Mono.defer(() -> context.appendMessage(compactedMessages)))
                        .doOnSuccess(v -> wire.send(new CompactionEnd()))
                        .doOnError(e -> wire.send(new CompactionEnd())));   // ← 失败也发 End
    }
    return Mono.empty();
}
```

**关键事实**：
- **触发阈值** = `maxContextSize - EngineConstants.RESERVED_TOKENS`（RESERVED_TOKENS 是一个预留 token 数常量，避免压缩恰好撞线）
- **回退到 checkpoint 0**（`context.revertTo(0)`），然后 append 压缩后的消息——这就是为什么 `/reset` 和 `/compact` 都用 `checkpointCount > 0` 作为"是否有东西可清/压"的判断。
- **Wire 事件**：`CompactionBegin` / `CompactionEnd` 由 `WireMessageHandler` 消费（09 篇 §7.6 已讲），在 UI 上显示 🗜️ 状态。
- **异常路径也发 `CompactionEnd`**——确保 UI 不会卡在 "Compacting" 状态。

**调用方**：`AgentExecutor` 在每步开始前调用 `contextManager.checkAndCompact(...)`，所以每一步都会检查一次。

### 4.4 `checkpointCount` 的真实含义

`ContextQueryRequest.ContextInfo.checkpointCount` 由 `WireRequestHandler#handleContextQuery`：

```java
.checkpointCount(context.getnCheckpoints())
```

**关键纠正**：`Context` 类（`core/engine/context/Context.java`）内部**不维护 checkpoint 列表**，只有一个 `int nextCheckpointId` 计数器。真正的 checkpoint 以 **磁盘 JSONL 的 `_checkpoint` 元数据行** 形式持久化（`{"role": "_checkpoint", "id": N}`）。方法签名是 `Mono<Integer> checkpoint(boolean addUserMessage)`——**不叫 `createCheckpoint`**。核心逻辑（`Context.java#162-180`）：

```java
public Mono<Integer> checkpoint(boolean addUserMessage) {
    int checkpointId = nextCheckpointId++;
    return repository.saveCheckpoint(checkpointId)           // 写 _checkpoint 行到磁盘
            .then(Mono.defer(() -> {
                if (addUserMessage) {                         // 仅当显式要求，才在历史里插一条可见消息
                    Message checkpointMessage = Message.builder()
                            .role(MessageRole.USER)
                            .content(List.of(TextPart.of("<system>CHECKPOINT " + checkpointId + "</system>")))
                            .build();
                    return appendMessage(checkpointMessage).thenReturn(checkpointId);
                }
                return Mono.just(checkpointId);
            }));
}
```

**Jimi 当前所有调用点都传 `addUserMessage=false`**（`AgentExecutor.java#107` 和 `#158`）——即 checkpoint 仅以磁盘元数据行形式存在，不在对话历史中插入可见消息。

**checkpoint 0 的真实时机**：`AgentExecutor#107` 的用户输入处理链路：

```java
return triggerHookSafely(HookType.USER_PROMPT_SUBMIT, userPromptHookContext)
        .then(context.checkpoint(false))                      // ← checkpoint 0 在这里诞生
        .then(context.appendMessage(userMessage))
        .then(context.updateTokenCount(context.getTokenCount() + userInputTokens))
        ...
        .then(contextManager.matchAndInjectTopics(...))        // Topic 按需注入
        .then(runReactLoop());
```

即 **checkpoint 0 是"第一次用户输入到达前"的状态**——此时已经走完 `restore()` 恢复历史（如果 `-C`）、已经触发 `USER_PROMPT_SUBMIT` hook，但用户的这条消息还没追加进 history。之后每步 ReAct 循环的 `beforeStep` 会再调 `checkpoint(false)`（`AgentExecutor#158`），产生 checkpoint 1, 2, 3...

`context.revertTo(checkpointId)` 的行为：**保留 `_checkpoint id=target` 行之前的所有内容**（含所有消息和更早的 checkpoint 行），丢弃之后的内容。所以：

- `revertTo(0)` = **回到第一次用户输入到达前的状态**——之前 `restore()` 恢复的历史（若有）仍保留，但本次启动后的所有用户输入、助手回复、工具调用、后续 checkpoint 都被丢弃。
- 注意：如果本次启动**不是** `-C`，`restore()` 恢复不到任何内容，此时 `revertTo(0)` 相当于清空 Context（除了 System Prompt——System Prompt 由 `JimiEngine` 在启动时 append，发生在 `checkpoint(false)` 之前还是之后决定了它是否被保留，这取决于具体启动流程）。
- 源码 JavaDoc "0 是第一个检查点" 与 `if (checkpointId >= nextCheckpointId)` 的守卫表明：`revertTo(0)` 要求 `nextCheckpointId >= 1`——即**至少发生过一次用户交互**。`/reset` 的 `ResetCommandHandler` 在 `checkpointCount == 0` 时直接返回"上下文为空，无需重置"，就是为了避免触发 `IllegalArgumentException`。

**`/reset` 与 `/compact` 判 `checkpointCount == 0`** 的真实语义：**"从未发生过任何用户交互"**——而不是"消息数为 0"。

### 4.5 Topic 按需注入

`ContextManager#matchAndInjectTopics`（Layer 2 的真实调用路径）：

```java
String userQuery = extractUserQuery(context);           // 取最后一条 USER 消息的 TextPart 拼接
MemoryStore store = memoryManager.getOrCreateStore(workDirPath);
TopicMatcher matcher = new TopicMatcher(store);
List<TopicMatcher.MatchedTopic> matchedTopics = matcher.match(userQuery, 2);   // 最多 2 个
if (matchedTopics.isEmpty()) return Mono.empty();
String topicContent = matcher.loadMatchedTopics(matchedTopics);
Message topicMessage = Message.builder()
        .role(MessageRole.ASSISTANT)                    // 以 ASSISTANT 角色（非 SYSTEM）
        .content(List.of(TextPart.of("[Memory Context] " + topicContent)))
        .build();
return context.appendMessage(topicMessage);
```

**观察**：
- **用 ASSISTANT 角色**注入 topic 而不是 SYSTEM——与 `SimpleCompaction` 的选择一致，是一种与 LLM 对多 SYSTEM 消息不友好的妥协。
- **最多 2 个 topic**（硬编码 `matcher.match(userQuery, 2)`）。
- 前缀 `"[Memory Context] "` 作为 LLM 识别该消息特殊性的标记。
- `TopicMatcher` 的具体匹配算法见该类源码（简单关键词匹配，非向量检索）。

### 4.6 Skill 摘要注入（相邻但独立）

`ContextManager#getSkillsSummary()` 返回 `skillRegistry.generateSkillsSummary()`，内容被 System Prompt 构建路径使用。**注意**：`matchAndInjectSkills(context, stepNo)` 方法已经**废弃**（返回 `Mono.empty()`），源码注释写明 "渐进式披露模式：不再自动匹配和注入技能"——改为在 System Prompt 提供摘要 + LLM 通过 `SkillsTool` 主动加载（06 篇详细讲）。

## 5 · `MemorySearcher` 与 `MemoryTool`

### 5.1 `MemorySearcher` 的三种搜索入口

源码 `memory/MemorySearcher.java`：

| 方法 | 签名 | 扫描范围 | 匹配方式 |
|---|---|---|---|
| `search` | `(Path sessionsDir, String query, int maxResults)` | 单个 `sessionsDir` 下的所有 `.jsonl` | 大小写不敏感子串 `contains` |
| `searchAll` | `(String query, int maxResults)` | `~/.jimi/sessions/**/*.jsonl`（**全部工作目录**） | 同上 |
| `searchRegex` | `(Path sessionsDir, String regex, int maxResults)` | 单个 `sessionsDir` 下的所有 `.jsonl` | `Pattern.compile(regex, CASE_INSENSITIVE).matcher(text).find()` |

**调用者**：
- `MemoryCommandHandler#searchHistory` 调 `searchAll`（5 条上限）。
- `MemoryTool#handleSearch` 也调 `searchAll`（5 条上限）——**两个入口都不是 `search(sessionsDir)`，因此参数层面无法限定搜索范围**，永远搜全部工作目录。

### 5.2 搜索文件排序

源码内对所有待扫描 `.jsonl` 按**最后修改时间倒序**排列：

```java
allJsonlFiles.sort((a, b) -> Files.getLastModifiedTime(b).compareTo(Files.getLastModifiedTime(a)));
```

即**越新的会话越先搜，命中 `maxResults` 就停**——老会话可能永远不会被扫到。默认 `DEFAULT_MAX_RESULTS = 5`（源码常量）。

### 5.3 JSONL 行解析

每行是一个独立 JSON 对象，字段读取方式：

```java
JsonNode contentNode = node.get("content");
if (contentNode.isTextual())      return contentNode.asText();
if (contentNode.isArray()) {
    StringBuilder textBuilder = new StringBuilder();
    for (JsonNode part : contentNode) {
        JsonNode textNode = part.get("text");
        if (textNode != null && textNode.isTextual()) {
            if (!textBuilder.isEmpty()) textBuilder.append(" ");
            textBuilder.append(textNode.asText());
        }
    }
    return textBuilder.isEmpty() ? null : textBuilder.toString();
}
```

即只提取 `content` 为字符串或数组中 `text` 字段——**`ToolCallPart` 的工具调用参数/结果不会被搜到**，因为它们序列化为非 text 结构。

### 5.4 片段切割规则

`createSnippet`：

- 文本 ≤ `CONTEXT_SNIPPET_MAX_LENGTH` → 原文返回
- 否则：定位 query 位置 → 前 50 字符 + query + 后 200 字符，两端加 `"..."`

### 5.5 `SearchResult` 的输出格式

`record SearchResult(String sessionId, int lineNumber, String role, String snippet)`。`format()` 把 sessionId 截断为前 8 字符（UUID 首段），role 映射为中英混合标签：`user → [User]`、`assistant → [Assistant]`、`system → [System]`、其他 `[<原名>]`。最终格式：

```
Session: abc12345 (line 42) [User]
  ...你的消息片段...
```

### 5.6 `TopicMatcher` 的匹配算法

源码 `memory/TopicMatcher.java`：

```java
int score = 0;
String[] keywords = topicName.toLowerCase().split("[-_]");   // 文件名按 '-' 和 '_' 分词
for (String keyword : keywords) {
    if (keyword.length() < 2) continue;                       // 跳过 1 字符短词
    if (inputLower.contains(keyword)) score += keyword.length();
}
return score;
```

**关键事实**：
- 打分粒度为**关键词字符长度之和**——匹配到 `"project-structure"` 的两个关键词 `project` 和 `structure` 分别加 7 和 9，总分 16。
- **用户输入侧不分词**，直接用原文 `contains` 判断——`"projekt"`（拼错）不会命中。
- 分数降序排序后取前 `maxTopics` 个（`ContextManager` 调用传 2）。
- `loadMatchedTopics` 把每个 topic 拼成一段：`## Relevant Memory Topics\n\n### <Topic Name>\n\n<file content>\n\n`——`formatTopicName` 把 `project-structure` 美化为 `Project Structure`（首字母大写 + 空格分隔）。
- **不依赖向量/LLM**，纯字符串包含匹配——Topic 文件名起得好不好直接决定命中率。

### 5.7 `MemoryTool`（Agent 主动调用的工具）

源码 `tool/core/MemoryTool.java` 是一个 `SyncTool<Params>`，`@Scope(SCOPE_PROTOTYPE)` 表示**每次 ToolRegistry 组装都 new 一个**。它通过三个 setter 接收运行时依赖：`setMemoryManager` / `setWorkDirPath` / `setSessionsDir`——这些在 `ToolRegistryFactory` 装配路径上注入。

支持 7 种 action（对应 JSON param 的 `action` 字段）：

| action | 必填参数 | 底层调用 |
|---|---|---|
| `read` | — | `memoryManager.readMemory(workDirPath)` |
| `write` | `section`, `content` | `memoryManager.writeMemory(workDirPath, section, content)` |
| `append` | `section`, `content` | `memoryManager.appendMemory(workDirPath, section, content)` |
| `search` | `query` | `new MemorySearcher().searchAll(query, 5)` |
| `list_topics` | — | `store.listTopics()` |
| `read_topic` | `topic_name` | `store.readTopic(topicName)` |
| `write_topic` | `topic_name`, `content` | `store.writeTopic(topicName, content)` |

**注意**：
- `search` 的 `sessionsDir` 成员**传了但未被 `handleSearch` 使用**——`handleSearch` 直接调 `searchAll` 扫全局（与 `MemoryCommandHandler` 一致）。`sessionsDir` 成员字段形同虚设。
- **未支持 delete/clear_topic**——Agent 只能通过 `write_topic` 覆盖写入，无法删除 topic 文件。`/memory clear` 也只是把 MEMORY.md 写空串，不删除 topics。
- `isConcurrentSafe() == false` —— 告诉 ToolRegistry 该工具不可并发执行（避免多次 append 竞争同一文件）。

### 5.8 `/memory` 命令 vs `MemoryTool`：两条并行路径

两者都写同一个 `MEMORY.md`，但**触发者不同**：

| 维度 | `/memory` 命令 | `MemoryTool` |
|---|---|---|
| 触发者 | **人类**（在 REPL 输入 `/memory ...`） | **Agent**（LLM 在对话中决定调用工具） |
| 入口 | `MetaCommandProcessor` → `CommandRegistry` → `MemoryCommandHandler` | ReAct 循环 → `executeTool("Memory", args)` |
| 审批 | 无 | 受 `ApprovalManager` 策略控制（yolo 模式下直通） |
| 批量 | 按子命令一次一个操作 | 一次一 action，但 Agent 可连续多次调用 |

这两条路径底层都走 `MemoryManager`，因此**两者写入的记忆在同一个 `MEMORY.md` 里混合**——从 MEMORY.md 本身看不出某条是人工写的还是 Agent 写的。

## 6 · 启动时的 Session 恢复

### 6.1 `--continue` CLI 参数

源码 `CliApplication.java#73-76`：

```java
@Option(names = {"-C", "--continue"}, description = "Continue the previous session")
private boolean continueSession;
```

**关键事实**：短参是 `-C`（**大写**），不是常见的 `-c`。`-c` 在 Jimi 中不是 `continue` 的别名——敲 `jimi -c` 会报 unknown option。

### 6.2 启动时的分支逻辑

源码 `CliApplication.java#132-` 的 session 装配：

```java
Session session;
if (continueSession) {
    Optional<Session> existingSession = sessionManager.continueSession(workDir);
    if (existingSession.isPresent()) {
        session = existingSession.get();
        System.out.println("✓ Continuing previous session: " + session.getId());
    } else {
        System.err.println("No previous session found for the working directory");
        session = sessionManager.createSession(workDir);
    }
} else {
    session = sessionManager.createSession(workDir);
}
```

**观察**：
- 默认行为（不带 `-C`）=**永远创建新 session**。
- `-C` 失败时（无历史 session 或 workDir 未注册过）会 `System.err` 一条消息后**静默降级为新建**——不终止启动。
- 终端会看到 `✓ Continuing previous session: <UUID>` 打印在标准输出，`--continue` 成功时的唯一可视反馈。

### 6.3 恢复后的 Context 加载

**关键纠正**：`Context` 的构造器（`core/engine/context/Context.java#63-69`）**不会自动读 `.jsonl`**——它只初始化一个空的 `history` List 和一个 `JSONLContextRepository`（`this.repository = new JSONLContextRepository(fileBackend, objectMapper)`）。真正的历史加载发生在 **`JimiFactory#262`** 的"第 10 步"：

```java
// JimiFactory#create(...) 的末尾片段
// 10. 恢复上下文历史 + 触发 SESSION_START hook
return context.restore()
        .then(Mono.defer(() -> {
            if (hookRegistry != null) {
                HookContext hookContext = HookContext.builder()
                        .hookType(HookType.SESSION_START)
                        .workDir(jimiRuntime.getWorkDir())
                        .sessionId(session.getId())
                        .build();
                return hookRegistry.trigger(HookType.SESSION_START, hookContext)
                        .onErrorResume(e -> { log.warn(...); return Mono.empty(); });
            }
            return Mono.empty();
        }))
        .then(Mono.just(soul));
```

**`Context#restore()`**（`Context.java#79-97`）委托 `repository.restore()`，把每条消息加回 `history`、恢复 `tokenCount` 和 `nextCheckpointId`：

```java
public Mono<Boolean> restore() {
    return repository.restore()
            .map(restoredContext -> {
                history.addAll(restoredContext.getMessages());
                this.tokenCount = restoredContext.getTokenCount();
                this.nextCheckpointId = restoredContext.getNextCheckpointId();
                boolean hasData = !restoredContext.getMessages().isEmpty();
                if (hasData) log.info("Restored context: {} messages, ...", ...);
                return hasData;
            })
            .onErrorResume(e -> { log.error(...); return Mono.just(false); });
}
```

**关键事实**：
- **`restore()` 每次启动都调用**——无论是否带 `-C --continue`。因为 `createSession` 返回的也是一个合法 `historyFile`，只不过是空文件，`restore()` 读它得到 `hasData=false`，不影响行为。
- `-C` 的真正作用**不是决定"是否 restore"**，而是决定 `session.historyFile` 指向**旧 UUID 的 `.jsonl`** 还是**新创建的空 `.jsonl`**——两种情况下 `restore()` 都会执行，但前者恢复出有内容的历史，后者恢复出空历史。
- `restore()` 的返回值 `Boolean hasData` 当前**在 JimiFactory 中被忽略**（`.then(Mono.just(soul))` 直接丢弃），没有基于"是否恢复成功"的分支逻辑。
- `onErrorResume` 兜底：即使 `.jsonl` 损坏或不存在，`restore()` 也返回 `false` 而非抛错——启动不会因此失败。

`Context#revertTo` 的 JavaDoc：

> "原文件会被轮转保存"

`JSONLContextRepository#revertToCheckpoint`（源码 `JSONLContextRepository.java#152-209`）的真实实现分三步：

1. **`Files.move(fileBackend, rotatedPath, REPLACE_EXISTING)`** —— 把 `abc.jsonl` 整体重命名为 `abc.jsonl.{N}`（序号由 `getNextRotationPath()` 分配）。
2. **逐行读 `rotatedPath` + 写 `fileBackend`（新建）** —— 读旧文件每行 JSON，遇到目标 `_checkpoint` 行就 `break`；否则把该行原样写入新 `abc.jsonl`。
3. 过程中同时把非元数据行 parse 成 `Message` 返回给调用方，恢复到内存 `history`。

**轮转命名规则**（`getNextRotationPath`）：

```java
for (int i = 1; i < 1000; i++) {
    Path rotatedPath = parent.resolve(baseName + "." + i);
    if (!Files.exists(rotatedPath)) return rotatedPath;
}
throw new IOException("No available rotation path found");
```

即 **`abc.jsonl.1` → `abc.jsonl.2` → ... → `abc.jsonl.999`**，**不是时间戳**。超过 999 次回退会抛 `IOException` 导致 `/reset` 失败——这是一个硬编码上限（见 §7.4 M-H13）。

**JSONL 文件内部还有元数据行**（源码注释+restore 实现）：

| 行类型 | 格式 | 作用 |
|---|---|---|
| 消息 | 标准 `Message` JSON | 对话内容 |
| 使用量 | `{"role": "_usage", "token_count": N}` | `restore` 时恢复 `tokenCount` |
| 检查点 | `{"role": "_checkpoint", "id": N}` | `revertToCheckpoint` 时读到目标 id 就截断 |

这意味着 `Context.getnCheckpoints()` 返回的 `nextCheckpointId` 是**从磁盘 `_checkpoint` 行数+1 重建出来的**，不是内存计数器——`restore()` 之后自动对齐。

这使得 `/reset` 和 `--continue` 之间的互动是：

1. 会话 A 写入 `abc.jsonl`，内容 100 行（含若干 `_checkpoint` 行）。
2. 用户 `/reset` → `revertTo(0)`：`abc.jsonl` 轮转为 `abc.jsonl.1`（若已存在则 `.2`），新的 `abc.jsonl` 只含旧文件中 **位于 `_checkpoint id=0` 之前** 的消息（通常是 System Prompt 相关内容）。
3. 下次 `--continue` 时，`metadata.json` 的 `last_session_id` 仍指向 `abc`，`continueSession` 返回的 Session `historyFile = abc.jsonl`——即**被 reset 过的版本**，而不是 reset 前的 100 行。轮转后的 `.1/.2/...` 文件永不被自动读取，只能手工检查。

### 6.4 SessionManager 的 AGENTS.md 预加载副作用

`SessionManager` 持有一个 `ConcurrentMap<String, String> agentsMdCache`（按工作目录绝对路径缓存），`public String loadAgentsMd(Path workDir)` 用 `computeIfAbsent` 实现读一次缓存一次。`createSession` 末尾的一行 `loadAgentsMd(workDir)` 就是**启动时的预热**——如果项目根有 `AGENTS.md`（优先大写）或 `agents.md`（兜底小写），文件内容会被同步读入缓存。`AGENTS.md` 的具体内容用途见 03 篇 §2。

**观察**：
- `continueSession` **不调 `loadAgentsMd`** —— 只有 `createSession` 路径会预热缓存。这意味着带 `-C --continue` 恢复的会话**如果之前没预热过**，第一次查询 AGENTS.md 会懒加载。
- 缓存 key 是 `workDir.toAbsolutePath().toString()`，**不经 hashCode**——与 Memory/Session 的路径哈希体系无关，直接用字符串比较。

### 6.5 未实装的"继续选择任意历史 session"能力

`WorkDirMetadata.sessionIds` 是一个 **历史 session ID 列表**（按创建顺序追加），但 `continueSession` **只读 `lastSessionId`**。没有命令或参数允许用户"从 session 列表挑一个恢复"——即使元数据里存着 N 个历史会话，也永远只能 resume 最后一个。要访问更早的会话，只能手工找到 `.jsonl` 文件用 `grep` 看，或通过 `/memory search` 搜关键词。

| M-L1 | `/compact` | "压缩上下文" | 见 09 篇 §8 L1：**不触发任何真实压缩**，真正的压缩在 `ContextManager#checkAndCompact` 自动触发 |
| M-L2 | `MemoryExtractor.extractUserIntent(context)` | 从 Context 提取用户意图 | 方法已实现（读最后一条 USER 消息），**但 `extract(...)` 主干从未调用它**——提取出的意图被静默丢弃 |
| M-L3 | `detectSection` 的 `User Preferences` / `Lessons Learned` | 分类条目到不同 section | 源码 `MemoryExtractor#detectSection` 有 4 个分支：含 `教训/lesson/注意/avoid` → `Lessons Learned`；含 `决策/decision/选择/chose` → `Key Decisions`；含 `modified/修改` → **`Key Decisions`**（而不是独立的 File Changes）；**其他默认 → `Key Decisions`**。**没有任何分支写入 `User Preferences`** —— 该 section 由 `initializeIfAbsent` 预创建但 `MemoryExtractor` 永远不会自动填充，只能通过 `/memory write "User Preferences"` 或 `MemoryTool` 手写 |
| M-L4 | `MemoryTool.sessionsDir` setter | 限定搜索范围 | setter 接收值但 `handleSearch` 用 `searchAll` 全局搜，该字段**从未被读取** |
| M-L5 | `WorkDirMetadata.sessionIds` 历史列表 | 似乎支持"挑选历史 session 恢复" | 无命令/参数消费该列表；`continueSession` 只读 `lastSessionId` |
| M-L6 | `SessionManager.continueSession` | 恢复上次会话 | **不检查 `historyFile` 是否存在**；若用户手工删了 `.jsonl`，仍返回指向幽灵文件的 Session |
| M-L7 | `SimpleCompaction` 保留条数 | 不可配置 | `MAX_PRESERVED_MESSAGES = 2` 硬编码——即无论 max_context_size 多大，压缩后只保留最后 2 条 USER/ASSISTANT 消息 |
| M-L8 | `MemoryConsolidator` LLM 校验 | 宽松校验 | 仅检查 `contains("# Project Memory")`，LLM 可以返回头部正确但内容任意的文本——**防御面窄** |
| M-L9 | `MemoryConsolidator` 规则整理的"去重" | 暗示去重 | 实际只保留前 `maxMemoryTokens/10` 条 `- ` 开头的条目（按原顺序），**未去重**，只是截断 |

### 7.2 命名/风格分歧

| # | 位置 | 分歧 |
|---|---|---|
| M-S1 | `Context.getnCheckpoints()` | 方法名 `getnCheckpoints`（小写 n）不符合 Java camelCase 规范；应为 `getNumCheckpoints` 或 `getCheckpointCount` |
| M-S2 | CLI 参数 `-C, --continue` | 大写 `-C`，与常见约定（多用 `-c`）不同；`-c` 在 Jimi 中不是该参数的别名 |
| M-S3 | Memory 三层 / Session / Context 的持久化根 | 都用 `~/.jimi/{子目录}/{hex(hashCode)}/`，但三个子目录**对同一 workDir 的 hex 值完全一致**——设计上看似独立实则耦合（若 `hashCode` 算法版本变更，三个目录一起失效） |
| M-S4 | `/memory` 命令的 `clear` | 只清 MEMORY.md，不动 topics/ 目录 |
| M-S5 | `MemoryTool` 的操作动词 | `read`/`write`/`append`/`search`/`list_topics`/`read_topic`/`write_topic`——**snake_case 与 camelCase 混用**（单词 + 带下划线），且 read_topic 的反向操作不是 `delete_topic`（未提供） |

### 7.3 预留未实装

| # | 位置 | 预留情况 |
|---|---|---|
| M-P1 | `MemoryStore.deleteTopic` | 实现存在、被 `/memory clear` 不使用、`MemoryTool` 也不暴露对应 action——**只能通过编程调用，人/Agent 两条路径都摸不到** |
| M-P2 | `MemoryExtractor.extractUserIntent` | 实现了但主干不调用 |
| M-P3 | `MemoryStore.appendToMemoryMd` | 以"整段追加"方式拼接 `existing + "\n\n" + section`——`MemoryManager` 不使用它，只在测试或未来路径可能调用 |
| M-P4 | `MemorySearcher.search(sessionsDir)` 与 `searchRegex` | 两个方法签名都暴露了 `sessionsDir` 参数，但所有调用者都走 `searchAll`（无 sessionsDir 限定） |
| M-P5 | `MemoryStore.customStoragePath` | `MemoryConfig.storagePath` 非空时所有 workDir 共享同一个 memory root——暗示"全局记忆"模式，但没有任何 UI/doc 说明该配置 |
| M-P6 | `MemoryConsolidator.LOCK_FILE = "consolidation.lock"` | 常量声明了但**从未被任何代码读/写**——真实的并发控制是 `ConcurrentMap<String, AtomicBoolean> consolidationLocks`（**内存级**，**非跨进程**）。若未来多个 Jimi 进程并发整理同一工作目录，会绕过这个内存锁触发 race condition |
| M-P7 | `MemoryExtractor.extractFileModifications` 的 `files.size() < 2` 阈值 | 表面是"仅当修改多个文件时记录"，实际意味着**"只修改 1 个文件的执行不会被记忆提取"**——这是个容易被忽略的筛选条件，源码仅在代码注释中提及 |

### 7.4 硬编码简化

| # | 位置 | 硬编码项 |
|---|---|---|
| M-H1 | `EngineConstants.RESERVED_TOKENS` | `20_000` 写死，用户无法在 `config.yml` 调整触发压缩的阈值 |
| M-H2 | `SimpleCompaction.MAX_PRESERVED_MESSAGES` | `2` 写死 |
| M-H3 | `SimpleCompaction.COMPACT_TEMPLATE` | 中文 prompt 写死在源码 |
| M-H4 | `MemoryManager.formatForSystemPrompt` | 英文前缀段写死，不随用户语言切换 |
| M-H5 | `MemoryStore.initializeIfAbsent` | MEMORY.md 默认模板英文 + 4 个固定 section，不可定制 |
| M-H6 | `MemoryExtractor.detectSection` | 中英关键词清单写死，且 `修改` 归为 `Key Decisions` 而非独立 `File Changes` section |
| M-H7 | `MemorySearcher.DEFAULT_MAX_RESULTS` / `CONTEXT_SNIPPET_MAX_LENGTH` | 写死；`/memory search` 和 `MemoryTool.search` 都传 5 |
| M-H8 | `MemoryConsolidator.CONSOLIDATION_PROMPT` | 写死中文 prompt，含"不超过 2000 tokens"的提示 |
| M-H9 | `MemoryConsolidator.regex "# Project Memory"` | 作为 LLM 响应的有效性校验——若 LLM 用 `# 项目记忆` 或其他变体头部则整理被丢弃 |
| M-H10 | `ContextManager.matchAndInjectTopics` 的 2 | `matcher.match(userQuery, 2)` —— 最多注入 2 个 topic，不可配置 |
| M-H11 | `TopicMatcher.keyword.length() < 2` 过滤 | 1 字符短关键词永远不参与匹配 |
| M-H12 | Session / Memory 路径哈希 | `Integer.toHexString(String.hashCode())`—— 32 位有碰撞风险，虽然实际碰撞概率极低 |
| M-H13 | `JSONLContextRepository.getNextRotationPath` 的 `< 1000` 上限 | `/reset` 后产生 `.1` ~ `.999` 共 999 个轮转文件；第 1000 次回退抛 `IOException` 导致 `/reset` 失败，且没有自动清理老轮转文件的机制 |

### 7.5 与 Claude Code 对标的差距

本项目多处 JavaDoc 明示"对标 Claude Code 的 xxx"（Layer 1/2/3、autoDream、Manual write 等概念），归档如下差距，便于未来对齐：

| # | Claude Code 概念 | Jimi 当前实现 | 差距 |
|---|---|---|---|
| M-G1 | Layer 2 按需加载相关知识 | `TopicMatcher` 关键词匹配 | Claude Code 使用语义检索；Jimi 只用文件名关键词 `contains` |
| M-G2 | autoDream 后台整理 | `MemoryConsolidator.shouldConsolidate` + `triggerConsolidation` | Jimi 的 "后台" 是 `.subscribe()` 异步触发，整理过程阻塞同一 LLM 连接（无单独的 reflection model） |
| M-G3 | Manual write（用户说"记住 X"即写入） | `MemoryTool`（Agent 侧）+ `/memory write`（人侧）两条路径 | 没有从用户自然语言自动触发写入的解析层——用户必须显式用 `/memory write` 或引导 Agent 调用 `MemoryTool` |
| M-G4 | 记忆去重与冲突消解 | 无 | autoDream 规则分支只截断不去重；LLM 分支依赖提示词"去除重复" |

## 8 · 一句话总结

Jimi 的"记忆 + 会话 + 上下文压缩" 形成**三个独立但通过 `~/.jimi/{hex(workDir)}/` 路径规则共享标识**的子系统：**Session** 把每次对话持久化为 JSONL（含 `_usage` / `_checkpoint` 元数据行）、靠 `metadata.json` 记录 `last_session_id`；**Context** 在 `JimiFactory#262` 启动时必调 `restore()` 把 `.jsonl` 加载回内存，运行期用 `Context.getnCheckpoints()` 跟踪检查点数量，Token 超阈值（`maxContextSize − EngineConstants.RESERVED_TOKENS(20_000)`）时由 `ContextManager#checkAndCompact` 在 `AgentExecutor` 的每步 `beforeStep` 自动调 `SimpleCompaction`（保留最后 2 条消息 + LLM 总结早期历史），`/reset` 触发的 `revertToCheckpoint(0)` 会把旧 `.jsonl` 轮转为 `.1`/`.2`/...`.999`；**Memory** 通过 `MemoryManager` 以 `MEMORY.md` + `topics/*.md` 的三层 Markdown 形态跨会话持久化，每轮 Agent 成功（非 sub-agent）后由 `MemoryExtractor` 从 `ExecutionState.keyDecisions/lessonsLearned/filesModified` 抽取条目并 `appendMemory`，达到 `sessionCount ≥ 5 且距上次 ≥ 24 小时` 后由 `MemoryConsolidator` 触发 `autoDream`（优先 LLM、无 LLM 则规则截断），`TopicMatcher` 在每次用户提问前用关键词包含打分匹配最多 2 个 topic 注入上下文；**但 `/compact` 命令不触发真实压缩、`MemoryExtractor` 不会自动填充 `User Preferences` section、`continueSession` 不校验文件是否真实存在、`SimpleCompaction` 硬编码保留 2 条、轮转文件数硬编码 999 上限、`LOCK_FILE` 常量声明却未使用** 等 9 处语义落差、5 处命名分歧、7 处预留未实装、13 处硬编码简化、4 处与 Claude Code 对标差距（共 38 条偏离），意味着任何基于本文档的二次开发都必须以源码为准。
