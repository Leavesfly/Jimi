# 13 · 插件系统与扩展分发

> **Plugin** 是 Jimi 把 **Skills / Hooks / Commands / MCP / Agents** 五种扩展点**打包成统一可分发单元**的机制。它解决的不是"如何写一个扩展点"（那是 12 篇的主题），而是"**如何把一组扩展点作为一个有版本号、有清单、可整体安装/卸载/升级的产品分发给团队**"。
>
> 本篇从源码视角拆解 `io.leavesfly.jimi.plugin` 包的全部实现，覆盖：清单规范、三层作用域、加载链路、模块适配器、安装器、兼容性校验、`/plugin` 命令与典型踩坑。

## §1 为什么需要插件系统

### 1.1 问题出发点

Jimi 在 12 篇之前已经有了五种扩展点：

| 扩展点 | 原生目录（用户级） | 规范形态 |
|---|---|---|
| **Skills** | `~/.jimi/skills/<name>/SKILL.md` | Markdown + YAML Front Matter |
| **Hooks** | `~/.jimi/hooks/*.yaml` | YAML 声明式 |
| **Commands** | `~/.jimi/commands/*.yaml` | YAML 声明式 |
| **MCP** | `~/.jimi/mcp/*.json` | JSON 配置 |
| **Subagent** | `~/.jimi/agents/<name>/agent.yaml` | YAML 声明式 |

没有插件机制之前，"Java 开发全家桶"这类**复合扩展**要用户把一堆文件**手动分发到 5 个不同目录**，升级和卸载极其繁琐——删一个 Skill 容易，把相关的 Hook、Command、MCP 配置一起干净地删掉就很麻烦。

### 1.2 插件解决的核心矛盾

插件把这 5 种扩展点**封装成一个有版本号、有清单、可整体分发的目录**（或 ZIP 归档），用 `/plugin install <source>` 一键安装、`/plugin uninstall <name>` 一键卸载。核心价值有四：

- **可分发**：一个 `plugin.yaml` + 五个可选子目录 = 一个独立产品
- **可审计**：`provides` 白名单显式声明对外暴露的扩展项，用户能审查到"这个插件到底注入了什么"
- **可版本化**：`compatibility.jimi_version` 做版本门禁，防止老插件跑在新 Jimi 上出怪问题
- **可灰度**：`defaults.modules` 模块级开关支持分阶段启用（例如先只上 Skill，Hook 下个版本再开）

### 1.3 与"扩展开发指南"的分工

| 维度 | 12 · 扩展开发指南 | 13 · 插件系统（本篇） |
|---|---|---|
| 定位 | 写**单个**扩展点（Tool / Skill / Hook / Command） | 把**一组**扩展点打包分发 |
| 产出物 | 一个 Java 类 / 一个 Markdown 文件 / 一个 YAML 文件 | 一个完整插件目录（含 `plugin.yaml`） |
| 读者 | 想给 Jimi 加一个小能力 | 想发布一个"Java 开发全家桶"给团队共享 |
| 源码包 | `tool.core` / `skill` / `core.hook` / `command.custom` | `io.leavesfly.jimi.plugin` |

**经验法则**：单个小能力用 12 篇的方式；一旦扩展点超过 2 个、或需要多人共享、或要发布到 GitHub，就应当升级为插件。

## §2 整体架构与核心组件

### 2.1 分层结构

插件系统在架构上是**一个独立的子模块**，通过"**零侵入 + 适配器**"的模式桥接到 Jimi 现有的各个 Registry，不改写任何核心类：

```
┌───────────────────────────────────────────────────────────────────┐
│  CLI 层                                                            │
│  PluginCommandHandler  ←─── /plugin list/info/enable/... 命令入口   │
└──────────────────────────────┬────────────────────────────────────┘
                               │
┌──────────────────────────────▼────────────────────────────────────┐
│  门面层                                                            │
│  PluginRegistry      ←─── 对外核心 API、状态缓存、生命周期管理     │
│    ├── @PostConstruct → 启动时加载 CLASSPATH + USER                │
│    ├── loadProjectPlugins(projectDir) → JimiFactory 启动时触发     │
│    └── enable / disable / reload / list / findByName               │
└──────────┬────────────────────────────────────┬───────────────────┘
           │                                    │
┌──────────▼──────────────┐    ┌────────────────▼──────────────────┐
│  加载器                  │    │  分发器                            │
│  PluginLoader            │    │  PluginDispatcher                  │
│   扫描三层目录           │    │   遍历所有 PluginModuleAdapter     │
│   + 解析 plugin.yaml     │    │   + 检查模块开关                   │
│   + 兼容性/环境校验      │    │   + 捕获单 Adapter 异常不中断其他  │
└──────────────────────────┘    └────────────────┬──────────────────┘
                                                 │
             ┌───────────────┬───────────────┬──┴──────────┬────────────────┐
             ▼               ▼               ▼             ▼                ▼
       ┌──────────┐   ┌──────────┐   ┌───────────┐   ┌─────────┐   ┌──────────┐
       │ Skills   │   │ Hooks    │   │ Commands  │   │ MCP     │   │ Agents   │
       │ Adapter  │   │ Adapter  │   │ Adapter   │   │ Adapter │   │ Adapter  │
       └────┬─────┘   └────┬─────┘   └─────┬─────┘   └────┬────┘   └────┬─────┘
            ▼              ▼               ▼              ▼             ▼
     SkillRegistry   HookRegistry  CustomCommandReg.  (discovery-only 登记)
       （真正注入）                                   供 JimiFactory 消费

┌───────────────────────────────────────────────────────────────────┐
│  安装器                                                            │
│  PluginInstaller ── install(source) 自动分派                       │
│    ├── installFromGitHub(owner/repo[#branch])  主/备分支 fallback  │
│    ├── installFromUrl(zipUrl)                                      │
│    └── installFromLocal(dir | .zip)                                │
│         底层复用 ArchiveInstaller（下载 / Zip-Slip 安全解压 / 复制）│
└───────────────────────────────────────────────────────────────────┘
```

### 2.2 两个核心设计约束

**① 零侵入：Adapter 只调用 Registry 的公开 API**

每个 `PluginModuleAdapter` 实现只依赖已有 Registry 的 `register` / `unregister` 这种**原有的公开方法**，不强迫 Registry 为插件系统新增专属接口。例如 `SkillModuleAdapter` 直接复用 `SkillLoader.loadSkillsFromDirectory()` + `SkillRegistry.register()`，`HookModuleAdapter` 直接用 `HookRegistry.registerHook()`。

好处是：**拔掉整个插件系统，Jimi 核心依旧正常工作**，这让插件系统可以独立演进。

**② 白名单保护：`provides` 显式声明对外扩展项**

即便插件目录下塞了一百个 `SKILL.md`，也只有 `plugin.yaml` 的 `provides.skills` 白名单内的会被注册。空白名单 = 全部放行（便于本地开发），非空白名单 = 严格过滤。规则统一由 `PluginProvides.allowsXxx()` 系列方法实现。

### 2.3 三层作用域

与 Skills / Hooks 一致，插件也遵循"**classpath < user < project**"的三层语义：

| 作用域 | 加载目录 | 优先级 | 典型来源 |
|---|---|---|---|
| `CLASSPATH` | `classpath:/plugins/<name>/` | 最低 | Jimi 官方内置（JAR 内） |
| `USER` | `~/.jimi/plugins/<name>/` | 中 | `/plugin install` 的默认落点 |
| `PROJECT` | `<project>/.jimi/plugins/<name>/` | 最高 | 随 git 提交、团队共享 |

**同名后加载覆盖**：如果用户级和项目级各有一个叫 `java-dev-kit` 的插件，项目级的 `provides` 白名单内扩展项会把用户级的覆盖掉（Registry 层面由各自的 `register` 方法决定同名处理策略）。

### 3.2 关键类职责一览

| 类 | 构造型 | 职责 | 关键方法 |
|---|---|---|---|
| `PluginRegistry` | `@Service` | 对外门面、状态缓存、生命周期编排 | `loadProjectPlugins` / `enable` / `disable` / `reload` / `list` |
| `PluginLoader` | `@Service` | 扫描目录、解析清单、做兼容性/环境校验 | `loadAllFromClasspath` / `loadAllFromUserHome` / `loadAllFromProject` / `checkCompatibility` / `checkRequirements` |
| `PluginDispatcher` | `@Component` | 把插件分发给各模块 Adapter | `dispatch(spec)` / `unload(previousResult)` |
| `PluginModuleAdapter` | interface | 单扩展点的桥接协议 | `getModuleName` / `supports` / `load` / `unload` |
| `PluginInstaller` | `@Service` | 从 GitHub / URL / 本地安装 | `install(source)` / `uninstall(name)` |
| `ArchiveInstaller` | `@Service` | 通用 ZIP 下载/解压/复制 | `downloadFile` / `unzipFile`（Zip Slip 防护） |
| `PluginCommandHandler` | `@Component` | `/plugin` 命令 | `list` / `info` / `enable` / `disable` / `reload` / `install` / `uninstall` / `doctor` |
| `VersionRange` | `static util` | 语义化版本范围匹配 | `matches(range, actual)` |

> **Spring 依赖顺序**：`PluginRegistry` 显式 `@DependsOn({"skillRegistry", "hookRegistry", "customCommandRegistry"})`，确保在插件做"追加注册"之前，三个原生 Registry 已完成自身的 `@PostConstruct` 初始化——**原生数据在前，插件追加在后**，避免启动顺序竞态。

## §4 `PluginSpec`：插件的数据模型

`PluginSpec` 是 `plugin.yaml` 反序列化后的**根 POJO**，贯穿加载、校验、分发、卸载全流程。本节按字段分组解剖。

### 4.1 最小清单与完整清单

```yaml
# ---------- 最小清单（三个必需字段） ----------
name: hello-world
version: 1.0.0
description: 我的第一个 Jimi 插件
```

`PluginSpec#validate()` 会在解析阶段强制要求 `name` / `version` / `description` 非空，否则抛 `IllegalArgumentException` 并在加载日志中留痕。

```yaml
# ---------- 完整清单 ----------
# 基本元信息
name: java-dev-kit
version: 1.2.0
description: Java 开发者全套扩展（Skill + Hook + Command）
author: "yefei.yf"
homepage: "https://github.com/leavesfly/jimi-java-dev-kit"
repository: "https://github.com/leavesfly/jimi-java-dev-kit"
license: "Apache-2.0"
keywords: [java, spring, development]

# 兼容性（在加载阶段校验，不满足则 reject）
compatibility:
  jimi_version: ">=0.1.0,<1.0.0"
  java_version: ">=17"
  os: [linux, mac]

# 运行环境依赖
requirements:
  binaries:
    - name: google-java-format
      check: "google-java-format --version"      # 自定义检查命令（可选）
      install_hint: "brew install google-java-format"
  env_vars:
    - JAVA_HOME
  files:
    - pom.xml                                    # 相对项目根目录

# 扩展点白名单（空列表 = 全部放行；非空 = 严格过滤）
provides:
  skills: [java-review]
  hooks: [auto-format-java]
  commands: [review]
  mcp_servers: [filesystem]
  agents: [Security-Agent]

# 默认启用状态 + 模块级开关
defaults:
  enabled: true
  modules:
    skills: true
    hooks: true
    commands: true
    mcp: true
    agents: true

# 插件间依赖（MVP 仅校验版本，不做传递加载）
dependencies:
  - name: git-toolkit
    version: ">=0.3.0"

# 生命周期脚本（字段已定义；脚本执行器在后续阶段实装）
lifecycle:
  on_install:   "scripts/post-install.sh"
  on_enable:    "scripts/on-enable.sh"
  on_disable:   "scripts/on-disable.sh"
  on_uninstall: "scripts/pre-uninstall.sh"
```

### 4.2 字段语义速查表

| 字段 | 类型 | 必填 | 说明 |
|---|---|:---:|---|
| `name` | `String` | ✓ | 插件唯一标识；同名插件后加载覆盖先加载 |
| `version` | `String` | ✓ | 语义化版本号；形如 `1.2.0` / `0.1.0-SNAPSHOT` |
| `description` | `String` | ✓ | 简短描述 |
| `author` / `homepage` / `repository` / `license` | `String` | 否 | 元信息，用于 `/plugin info` 展示与未来的插件市场检索 |
| `keywords` | `List<String>` | 否 | 检索关键词 |
| `compatibility` | `PluginCompatibility` | 否 | Jimi / Java / OS 兼容性约束（见 §10） |
| `requirements` | `PluginRequirements` | 否 | 必需的可执行命令、环境变量、项目文件 |
| `provides` | `PluginProvides` | 否 | 扩展点白名单；空列表 = 全放行 |
| `defaults` | `PluginDefaults` | 否 | 默认启用状态 + 模块级开关 |
| `dependencies` | `List<PluginDependency>` | 否 | 插件间依赖；MVP 仅做版本校验 |
| `lifecycle` | `PluginLifecycle` | 否 | 生命周期脚本；**MVP 阶段只存字段，执行器待实装** |
| `pluginDir` | `Path` | transient | 运行时由 Loader 填充的插件根绝对路径 |
| `scope` | `PluginScope` | transient | 运行时由 Loader 填充的来源作用域 |

### 4.3 `PluginProvides` 白名单的工作方式

`PluginProvides` 提供 5 个 `allowsXxx()` 方法，每个方法遵循**"空列表 = 全放行"** 语义：

```text
// PluginProvides#allowsSkill 的源码等效
boolean allowsSkill(String skillName)
    = skills == null || skills.isEmpty() || skills.contains(skillName);
```

这个设计让**本地开发阶段**可以不填 `provides`（全放行，便于迭代），**发布前**再加上白名单锁死对外暴露面。`SkillModuleAdapter` / `HookModuleAdapter` / `CommandModuleAdapter` 等在注册前都会调一次对应的 `allowsXxx`，不在白名单的会被过滤，日志中以 `DEBUG` 级别留痕。

### 4.4 `PluginDefaults` + `PluginModuleToggle` 的两层开关

- **插件级开关**：`defaults.enabled` 控制整个插件是否装载。设为 `false` 时 `PluginRegistry` 会记录为 `disabled` 状态但不调 Dispatcher。
- **模块级开关**：`defaults.modules.<module>` 控制插件内某一类扩展点是否加载。`PluginSpec#isModuleEnabled(name)` 的实现是**插件级 AND 模块级**，必须两级都为 `true` 才加载。

```text
// PluginSpec#isModuleEnabled 等效逻辑
isModuleEnabled(moduleName):
    if !isEnabled(): return false                    # 插件级优先
    return defaults.getModules().isModuleEnabled(moduleName)   # 模块级
```

这让作者可以做**灰度发布**：先发一个 `hooks: false` 的版本让用户熟悉 Skills 部分，下一版再把 Hooks 打开。

## §5 加载链路：从 `plugin.yaml` 到 `Registry`

### 5.1 启动期加载（CLASSPATH + USER）

`PluginRegistry` 用 `@PostConstruct` 在 Spring 上下文初始化完成后立即扫描 CLASSPATH 和 USER 两层：

```text
PluginRegistry#initialize（等效伪码）
  classpathPlugins = pluginLoader.loadAllFromClasspath()
  userPlugins      = pluginLoader.loadAllFromUserHome()
  for lp in classpathPlugins: dispatchAndRemember(lp)
  for lp in userPlugins:      dispatchAndRemember(lp)
```

PROJECT 层**不在此处加载**——项目根目录需要等到 `JimiFactory` 创建会话时才确定，由它显式调用 `pluginRegistry.loadProjectPlugins(projectDir)` 触发。代码里的实际触发点在 `JimiFactory` 第 9.1 步：

```text
JimiFactory（摘自源码，简化）
  if (pluginRegistry != null) {
      try {
          pluginRegistry.loadProjectPlugins(jimiRuntime.getWorkDir());
      } catch (Exception e) {
          log.warn("Failed to load project plugins: {}", e.getMessage());
      }
  }
```

### 5.2 `PluginLoader` 的扫描策略

`PluginLoader` 对不同作用域采用不同的扫描方式：

| 作用域 | 扫描方式 | 特殊处理 |
|---|---|---|
| `CLASSPATH`（JAR 模式） | 按 `BUILTIN_CLASSPATH_PLUGINS` 白名单逐一读取 | JAR 内部无法枚举子目录，必须硬编码名称；当前白名单**为空数组** |
| `CLASSPATH`（文件系统模式，开发环境） | 遍历 `classpath:/plugins/` 一级子目录 | 按 `TEST_FIXTURE_PLUGIN_NAMES`（当前含 `hello-world`）剔除测试夹具 |
| `USER` | 遍历 `~/.jimi/plugins/` 一级子目录 | 无特殊处理 |
| `PROJECT` | 遍历 `<projectDir>/.jimi/plugins/` 一级子目录 | `projectDir` 为 `null` 时直接返回空列表 |

JAR 模式判定：`ClassLoader.getResource("plugins")` 返回的 URL 协议等于 `"jar"` 时走白名单路径，否则用 `Paths.get(url.toURI())` 把资源 URL 转成文件系统路径后遍历一级子目录。

> **为什么 JAR 模式要白名单**：Java 的 ClassLoader 在 `jar:` 协议下不支持列目录，只能按已知资源路径读取（`getResourceAsStream`）。这是 Jimi 与 `SkillLoader.loadSkillsFromClasspath` 一致的做法——新增内置插件时需同时更新 `PluginLoader.BUILTIN_CLASSPATH_PLUGINS` 常量。

### 5.3 `LoadedPlugin`：含 rejected 的加载结果

`PluginLoader` 提供**两套**接口：

- `loadFromXxx()`：只返回校验通过的 `PluginSpec`（向后兼容，内部用 `filterAccepted` 过滤）
- `loadAllFromXxx()`：返回 `LoadedPlugin` record（`spec` + `rejectReason`），**包含**校验失败的插件

`LoadedPlugin` 是 `PluginLoader` 的内部 record，提供两个工厂方法：

- `LoadedPlugin.accepted(spec)` → `rejectReason == null`
- `LoadedPlugin.rejected(spec, reason)` → `isRejected() == true`

`PluginRegistry` 用的是后者，这样就能在 `/plugin list` 里把"装了但兼容性不过关"的插件显式标记为 ⛔ 拒绝，而不是静默丢弃——让用户知道"我装的这个为啥没生效"。

**解析失败 vs 校验失败的区别**：YAML 语法错或必需字段缺失时（`PluginSpec.validate()` 抛 `IllegalArgumentException`），该条目会被 `loadAllFromDirectory` 内部的 `try/catch` **直接丢弃并记 ERROR 日志**，不会进入 `LoadedPlugin.rejected` 列表——因为此时连可展示的 `PluginSpec` 都没有。只有"解析成功但校验失败"的条目才作为 `rejected` 返回。

### 5.4 `dispatchAndRemember`：四种状态机

`PluginRegistry#dispatchAndRemember` 是整个加载链路的**状态机中枢**，按以下顺序处理单个 `LoadedPlugin`：

```
┌────────────────────────────────┐
│ LoadedPlugin（来自 Loader）     │
└──────────────┬─────────────────┘
               │
  ┌────────────▼─────────────┐
  │ 1. loaded.isRejected()?  │  Loader 校验失败 → PluginState.rejected(reason)
  └────────────┬─────────────┘
               │ no
  ┌────────────▼─────────────┐
  │ 2. spec.isEnabled()==false? │  → PluginState.disabled（不调 Dispatcher）
  └────────────┬─────────────┘
               │ no
  ┌────────────▼─────────────┐
  │ 3. pluginDispatcher.dispatch(spec) │
  │    - 成功 → PluginState.enabled(spec, loadResult)
  │    - 抛异常 → PluginState.rejected("dispatch failed: " + msg)
  └──────────────────────────┘
```

整个流程**任何异常都不会向上抛出**——单个插件加载失败绝不能阻塞其他插件。这是插件系统的**稳定性基石**，源码用 `try/catch` 把 Dispatcher 异常兜底为 `rejected` 状态，并记入 `log.error`。

### 5.5 `PluginState`：运行时状态快照

每个已加载插件在 `PluginRegistry#loadedPlugins`（`ConcurrentHashMap<String, PluginState>`）里对应一个 `PluginState`：

| 字段 | 类型 | 含义 |
|---|---|---|
| `spec` | `PluginSpec` | 解析后的清单对象 |
| `enabled` | `boolean` | 当前是否启用 |
| `loadResult` | `PluginLoadResult` | 各模块的加载结果（仅启用时非 `null`） |
| `rejectReason` | `String` | 拒绝原因（仅 rejected 时非 `null`） |

对应三个工厂方法（`enabled` / `disabled` / `rejected`）形成了完整的状态组合。`/plugin list` 通过遍历这张表来展示状态图标：✅（enabled）/ ⏸️（disabled）/ ⛔（rejected）。

### 5.6 `reload` 的语义：重新从磁盘扫描

`PluginRegistry#reload()` **不是**"刷新内存中已有的 Spec"——它是**完整从磁盘重新扫描三层**：

```text
PluginRegistry#reload（等效伪码）
1. 反向卸载所有 enabled 插件的扩展点：
     for s in loadedPlugins.values():
         if s.enabled and s.loadResult != null: dispatcher.unload(s.loadResult)
2. 清空状态表：loadedPlugins.clear()
3. 按 CLASSPATH → USER → PROJECT 顺序重新扫描 + 分发：
     for lp in loader.loadAllFromClasspath(): dispatchAndRemember(lp)
     for lp in loader.loadAllFromUserHome():  dispatchAndRemember(lp)
     if currentProjectDir != null:
         for lp in loader.loadAllFromProject(currentProjectDir):
             dispatchAndRemember(lp)
```

这个语义确保了三件事：

- `/plugin install` 后 `reload` 能加载新装插件
- `/plugin uninstall` 后 `reload` 能彻底移除（不留"幽灵插件"）
- 用户在磁盘上手动改了 `plugin.yaml`，`reload` 能让改动立即生效

**PROJECT 层依赖 `currentProjectDir`（`AtomicReference<Path>`）缓存**，这样即便 `reload` 发生在 `loadProjectPlugins` 之后，也不会丢失项目级插件。如果先前从未调用 `loadProjectPlugins`（例如直接在无项目的交互中 `reload`），PROJECT 层会被跳过。

## §6 `PluginDispatcher` 与 `PluginModuleAdapter`

### 6.1 为什么需要 Adapter 层

"**插件内部有一堆 YAML/JSON/Markdown，Jimi 原生有一堆 Registry**"——中间需要一个翻译层。Jimi 选择的做法是定义一个统一的 SPI：

```text
interface PluginModuleAdapter
  String  getModuleName()                           # "skills" / "hooks" / ...
  boolean supports(Path pluginDir)                  # 插件里有对应子目录吗？
  ModuleLoadResult load(Path pluginDir, PluginSpec) # 扫 + 过滤 + 注册
  void    unload(PluginSpec, ModuleLoadResult prev) # 反注册（与 load 对称）
```

五个实现（`SkillModuleAdapter` / `HookModuleAdapter` / `CommandModuleAdapter` / `McpModuleAdapter` / `AgentModuleAdapter`）都是 `@Component`，Spring 会把它们作为 `List<PluginModuleAdapter>` 注入 `PluginDispatcher`，**加新扩展点只需要加一个 Adapter 类，无需改 Dispatcher**。

### 6.2 `PluginDispatcher#dispatch` 的执行循环

```text
PluginDispatcher#dispatch(spec)（等效伪码）
  results = LinkedHashMap<String, ModuleLoadResult>
  for adapter in adapters:                      # 保持 Spring 注入顺序
      module = adapter.getModuleName()
      if !spec.isModuleEnabled(module): continue   # 模块级开关
      if !adapter.supports(pluginDir):  continue   # 没有对应子目录
      try:
          results.put(module, adapter.load(pluginDir, spec))
      catch Exception e:
          results.put(module, ModuleLoadResult.failed(e))   # 单 Adapter 异常隔离
  return new PluginLoadResult(spec, results)
```

关键特性：

- **模块开关优先**：两级 AND 判定（插件级 + 模块级），任一为 `false` 都跳过
- **按需加载**：子目录不存在（`supports == false`）时跳过，**不做空目录扫描**
- **异常隔离**：每个 Adapter 都有独立 `try/catch`，一个模块炸了不影响其他
- **顺序可控**：按 Spring 注入顺序遍历，结果用 `LinkedHashMap` 保留顺序（影响 `/plugin info` 输出）

### 6.3 三个"真 · 注入型"Adapter 对比

`Skills` / `Hooks` / `Commands` 三个 Adapter **真正往对应的 Registry 里注册扩展项**，加载与卸载成对出现。三者实现结构高度相似：

| Adapter | 子目录 | 加载方式 | 注册调用 | 白名单方法 |
|---|---|---|---|---|
| `SkillModuleAdapter` | `<plugin>/skills/` | `SkillLoader.loadSkillsFromDirectory(dir, scope)` | `SkillRegistry.register(SkillSpec)` | `provides.allowsSkill(name)` |
| `HookModuleAdapter` | `<plugin>/hooks/*.yaml` | `YamlConfigLoader.loadFromDirectory(dir, HookSpec.class, source)` | `HookRegistry.registerHook(HookSpec)` | `provides.allowsHook(name)` |
| `CommandModuleAdapter` | `<plugin>/commands/*.yaml` | `YamlConfigLoader.loadFromDirectory(dir, CustomCommandSpec.class, source)` | `CustomCommandRegistry.registerCommand(CustomCommandSpec)` | `provides.allowsCommand(name)` |

**作用域映射**（仅 `SkillModuleAdapter` 需要）：

```text
插件作用域 → Skill 作用域
  PluginScope.PROJECT → SkillSpec.SkillScope.PROJECT
  CLASSPATH / USER    → SkillSpec.SkillScope.GLOBAL
```

这让**项目级插件**里的 Skill 自动承接 `PROJECT` 作用域，在 Skills 的三层优先级里也处于最高位，与插件自身的优先级对齐。

**source 标签**：`HookModuleAdapter` / `CommandModuleAdapter` 在调用 `YamlConfigLoader` 时会传入 `"plugin:" + spec.getName()` 作为来源标签，注册到 Registry 后可在 `/hooks list` / `/command list` 里看到具体是哪个插件贡献的——**可观测性**做在了最底层。

**覆盖语义**：`HookRegistry.registerHook` 与 `CustomCommandRegistry.registerCommand` 内部都会对同名条目先 `unregister` 再注册，因此多插件注册同名 Hook/Command 时，**后注册的会覆盖先注册的**（而加载顺序由 CLASSPATH → USER → PROJECT 决定，项目级最终胜出）。

### 6.4 两个"Discovery-Only"Adapter

`McpModuleAdapter` 和 `AgentModuleAdapter` **不真正往 Registry 注入**，只做"发现 + 登记"。原因是这两类扩展点的生命周期与其他三类不同：

**① MCP：`MCPToolProvider` 的 `mcpConfigFiles` 在 Engine 创建时一次性传入**，运行时不支持动态增删。所以 `McpModuleAdapter` 的做法是：

- 维护一张 `Map<String, List<Path>> discoveredConfigFilesByPlugin`（`ConcurrentHashMap`）
- 加载策略：**优先读 `servers.json` 单文件**；不存在时枚举所有 `*.json`
- 白名单按 `provides.mcp_servers` 过滤后，把 server 名列表写入 `ModuleLoadResult.loadedItems`

`PluginRegistry` 暴露两个查询接口给消费方：

- `getDiscoveredMcpConfigFiles(pluginName)` → 单插件视图（供 `/plugin info` 展示 `└─ 配置: <path>` 行）
- `getAllDiscoveredMcpConfigFiles()` → 全局视图（供 `JimiFactory` 合并后传给 `MCPToolProvider.setMcpConfigFiles`）

解析时的韧性：单个 `*.json` 解析失败仅 `log.warn`，不影响同一插件的其他 MCP 配置文件。

**② Agents：`AgentSpecLoader` 用 `@PostConstruct` 写死扫描路径**，动态追加需要改核心类。MVP 阶段选择"只发现不注入"——`AgentModuleAdapter.load` 用 `Files.walk(agentsDir)` 递归找所有 `agent.yaml`，以父目录名作为 agent 标识，按 `provides.agents` 白名单过滤后登记到 `ModuleLoadResult.loadedItems`，供 `/plugin info` 查看。`unload` 仅打 INFO 日志（`no-op in MVP`）。

> **`/plugin info` 如何区分两类 Adapter**：`PluginCommandHandler#isDiscoveryOnlyModule` 判定模块名 ∈ `{"mcp", "agents"}`。展示时：真注入的用 ✅，discovery-only 的用 🔍 并追加 `(discovery-only)` 标注，提示用户"这些扩展项仅被发现但未自动生效"。

### 6.5 `unload` 的对称性与幂等

每个 Adapter 的 `unload` 必须与 `load` **对称**——`load` 时往 Registry 注册了什么，`unload` 就反注册什么。这是 `disable` / `reload` 语义正确性的基础。以 `SkillModuleAdapter` 为例：

```text
SkillModuleAdapter#unload（等效伪码）
  for skillName in previousResult.getLoadedItems():
      try:
          if skillRegistry.unregister(skillName): removed++
      catch RuntimeException e:
          log.warn(...)    # 不抛，保证幂等
  log.info("unloaded {}/{} skill(s)", removed, total)
```

**幂等保证**：`unload` 时单个扩展项已不存在的情况不报错，`try/catch` 兜底为 `log.warn`。这是因为 `reload` 时可能先 `unload` 再 `dispatch`，中途磁盘状态可能已变化，不能让幂等性成为故障点。

**Discovery-only Adapter 的 unload**：由于没真正注入，`unload` 只清理内部登记表（如 `McpModuleAdapter` 的 `discoveredConfigFilesByPlugin.remove(name)`），避免 reload 时出现"幽灵配置"。

### 6.6 `ModuleLoadResult` 与 `PluginLoadResult` 数据模型

| 类型 | 字段 | 作用 |
|---|---|---|
| `ModuleLoadResult` | `List<String> loadedItems` + `Throwable error` | 单模块结果；`success(items)` / `failed(err)` 两个工厂 |
| `PluginLoadResult` | `PluginSpec spec` + `Map<String, ModuleLoadResult> moduleResults`（`LinkedHashMap`） | 单插件聚合结果 |

`PluginLoadResult` 额外提供两个便捷方法：

- `isAllSuccess()` —— 所有模块都成功
- `hasAnySuccess()` —— 至少一个模块成功

这两个方法目前未被命令层主动使用，但为未来的"插件健康度报告"留了扩展位。

## §7 安装与卸载：`PluginInstaller`

### 7.1 `install(source)` 的统一入口

`PluginInstaller#install(String)` 做自动分派：

```text
install(source):
  if source.contains("://")     → installFromUrl(source)       # HTTP(S) ZIP
  else if Files.exists(source)  → installFromLocal(source)     # 本地路径
  else                          → installFromGitHub(source)    # owner/repo[#branch]
```

三种来源的共同终点都是**复制到 `~/.jimi/plugins/<name>/`**（USER 层）。`PluginInstallResult` 三字段：`spec`（解析后的 `PluginSpec`）、`installedDir`（最终目录）、`source`（原始来源字符串，用于日志）。

安装完成后由调用方（`PluginCommandHandler.installPlugin`）触发 `pluginRegistry.reload()` 让新插件立即生效——**Installer 本身不触发 reload**，职责边界清晰。

### 7.2 GitHub 来源：主/备分支 fallback

GitHub 语法支持四种写法（优先级递减）：

```
owner/repo#branch                  # 显式分支
owner/repo/subpath#branch          # 显式分支 + 子路径（单仓库含多插件）
owner/repo                          # 隐式：按 DEFAULT_GITHUB_BRANCHES 顺序尝试
owner/repo/subpath                  # 隐式 + 子路径
```

底层下载地址模板：`https://github.com/%s/%s/archive/refs/heads/%s.zip`。

未显式指定分支时，`PluginInstaller` 按 `DEFAULT_GITHUB_BRANCHES`（`main` 优先、`master` 备选）顺序尝试。**只有 HTTP 404 才 fallback**，其他错误（401 / 403 / 500 等）直接包装为 `RuntimeException` 抛出——避免把网络/权限问题误判为分支问题。

ZIP 解压后会剥离 GitHub 归档自带的 `<repo>-<branch>/` 顶层包装目录（通过 `firstDirectChild` 找到唯一直接子目录），再按 `subPath` 进入具体子目录，然后用 `findPluginRoot` 找 `plugin.yaml`。

### 7.3 URL 与本地来源

- **`installFromUrl(url)`**：直接走 `doInstallFromZipUrl`，**无 fallback 语义**——任何 HTTP 非 2xx 响应都包装为 `RuntimeException` 抛出。
- **`installFromLocal(Path)`**：支持两种形态：
  - **目录**：用 `findPluginRoot` 递归（最多 `MAX_PLUGIN_ROOT_DEPTH = 3` 层深度）找含 `plugin.yaml` 的目录
  - **`.zip` 文件**：先 `ArchiveInstaller.unzipFile` 解压到临时目录，再找根

`MAX_PLUGIN_ROOT_DEPTH = 3` 的设计覆盖三种常见结构：根目录直有 `plugin.yaml`、GitHub 归档顶层包装一层、单仓库子目录放多个插件。深度 > 3 的被视为非合法归档，避免误匹配深埋文件。

### 7.4 原子切换：保证升级中途失败不损坏旧版

`copyToUserPlugins` 采用**三阶段原子切换**，是整个 Installer 里最精妙的部分：

```
~/.jimi/plugins/
├── <name>/         ← target：当前生效版本
├── <name>.new/     ← staging：新内容暂存区
└── <name>.bak/     ← backup：旧版本备份
```

执行顺序与回滚策略：

| 步骤 | 动作 | 失败后处理 |
|---|---|---|
| **0** | 清理历史残留的 `.new` / `.bak`（上次失败留的） | — |
| **1** | 把新内容复制到 `<name>.new` | 删半成品 `.new`，旧版原位保留 |
| **2** | 若旧版本存在，`<name>` → `<name>.bak` | 删 `.new`，旧版仍在原位 |
| **3** | `<name>.new` → `<name>`（`Files.move` with `REPLACE_EXISTING`） | 把 `.bak` 改回 `<name>`，同时清 `.new` |
| **4** | 清理 `<name>.bak`（降级为 warn，失败不影响可用性） | — |

这个设计保证：**任何中途失败都不会出现"旧版被删了、新版没装上"的砖头态**。`log.error("CRITICAL: failed to rollback...")` 是步骤 3 回滚失败时的最后护栏，提示用户手动清理。

### 7.5 `ArchiveInstaller` 的安全能力

`ArchiveInstaller` 是 `PluginInstaller` 和 `SkillsInstaller` 共用的**归档操作底座**，三个关键安全点：

- **Zip Slip 防护**：解压时对 `targetDir` 先 `toAbsolutePath().normalize()`，对每个 entry 也做 `normalize()`，最后用 `entryPath.startsWith(targetDir)` 校验——防止 `../../../etc/passwd` 类攻击路径逃出解压目录
- **HTTP 重定向层级限制**：最多 5 层（`MAX_REDIRECTS`），避免死循环；支持 301 / 302 / 303
- **强类型 HTTP 异常**：`HttpStatusException extends IOException` 把 `statusCode` 作为一等字段暴露，让 `PluginInstaller` 能基于 404 做确定性的分支 fallback，而不是脆弱的字符串解析

### 7.6 `uninstall(name)` 的约束

```text
uninstall(pluginName):
  target = userPluginsDir.resolve(pluginName)
  if !Files.isDirectory(target): return false
  archiveInstaller.deleteDirectory(target)
  return true
```

**只删 USER 层**——CLASSPATH 层在 JAR 内无法删除，PROJECT 层属于项目资产不应由插件命令代管。`PluginCommandHandler.uninstallPlugin` 在卸载失败时明确提示用户"**仅能卸载 `~/.jimi/plugins/` 下的用户级插件**"。调用方在 `uninstall` 成功后会触发 `pluginRegistry.reload()` 刷新状态。

## §8 `/plugin` 命令全集

`/plugin` 是 Jimi 暴露给终端用户的**管理入口**，由 `PluginCommandHandler`（`@Component`）实现。它实现 `CommandHandler` SPI，注册信息如下：

| 属性 | 值 |
|---|---|
| `getName()` | `"plugin"` |
| `getAliases()` | `["plugins"]` |
| `getCategory()` | `"plugin"` |
| `getPriority()` | `10` |
| `getUsage()` | `/plugin [list \| info <name> \| enable <name> \| disable <name> \| reload \| install <source> \| uninstall <name> \| doctor [<name>]]` |

### 8.1 子命令分派

`execute(CommandContext)` 用 Java 21 switch 表达式分派：

```text
subcommand = context.getArg(0)  // 无参数时调 listAllPlugins
switch subcommand:
  "list"              → listAllPlugins(out)
  "info" <name>        → showPluginDetails(name, out)
  "enable" <name>      → enablePlugin(name, out)
  "disable" <name>     → disablePlugin(name, out)
  "reload"             → reloadAll(out)
  "install" <source>   → installPlugin(source, out)
  "uninstall" | "remove" <name>
                        → uninstallPlugin(name, out)
  "doctor" [<name>]    → runDoctor(name|null, out)
  default              → showPluginDetails(subcommand, out)
                         # 作为 /plugin <name> 的便捷语法，等价于 info
```

**便捷语法**：`/plugin <name>` 相当于 `/plugin info <name>`——把第一个 arg 当作未知子命令时不报错，而是假设它是插件名去查详情。这是小工具的常见**渐进式 UX**。

### 8.2 子命令职责矩阵

| 子命令 | 委托对象 | 关键副作用 | reload 触发 |
|---|---|---|---|
| `list` | `PluginRegistry.list()` + `getStatistics()` | 无 | 否 |
| `info <name>` | `PluginRegistry.findByName()` + `getDiscoveredMcpConfigFiles()` | 无 | 否 |
| `enable <name>` | `PluginRegistry.enable()` → `PluginDispatcher.dispatch()` | 重新注册扩展点 | 否 |
| `disable <name>` | `PluginRegistry.disable()` → `PluginDispatcher.unload()` | 反注册扩展点 | 否 |
| `reload` | `PluginRegistry.reload()` | 三层磁盘重扫 | **是** |
| `install <source>` | `PluginInstaller.install()` + `PluginRegistry.reload()` | 落盘 + 重扫 | **是** |
| `uninstall <name>` | `PluginInstaller.uninstall()` + `PluginRegistry.reload()` | 删目录 + 重扫 | **是** |
| `doctor [<name>]` | `PluginLoader.checkCompatibility()` + `checkRequirements()` | 无 | 否 |

**异常处理**：`install` / `uninstall` 的 `try/catch` 把任何异常转为 `OutputFormatter.printError`，不让命令链路抛出——保证一个命令出错不会把整个 CLI 搞挂。

### 8.3 `/plugin list` 的输出样式

核心逻辑（`listAllPlugins` 简化）：

for state in plugins:
    icon = "⛔" if rejected else ("✅" if enabled else "⏸️")

### 8.4 `/plugin info <name>` 的区块结构

`showPluginDetails` 把单个插件的信息分为五块输出：

1. **基本信息**：名称、版本、描述、作用域、作者、许可、安装路径、状态
2. **兼容性**：`jimi_version` / `java_version` / `os`
3. **扩展点白名单**：`provides.{skills, hooks, commands, mcp_servers, agents}`，空列表展示为 `(未使用)`
4. **加载结果**：各模块的 `ModuleLoadResult`
   - 真注入模块（`skills`/`hooks`/`commands`）：✅ + 数量
   - Discovery-only（`mcp`/`agents`）：🔍 + 数量 + `(discovery-only)`
   - 失败模块：❌ + 错误消息
   - `mcp` 模块会追加 `└─ 配置: <path>` 行，列出 `getDiscoveredMcpConfigFiles(name)` 的结果
5. **插件依赖**：`dependencies` 列表

### 8.5 `/plugin doctor` 的诊断流程

`runDoctor(targetName, out)` 用来定位"为什么我的插件不加载"：

```text
runDoctor(target):
  jimiVersion = pluginLoader.getCurrentJimiVersion()
  targets = (target != null) ? [registry.findByName(target)] : registry.list()

  for each state in targets:
      compat = pluginLoader.checkCompatibility(state.spec, jimiVersion)
      reqs   = pluginLoader.checkRequirements(state.spec)
      print "✅ compatibility : OK"   or   "❌ compatibility : <reason>"
      print "✅ requirements  : OK"   or   "❌ requirements  : <reason>"
      if compat.passed() && reqs.passed():
          passed++, print "✅ OK"
      else:
          failed++, print "⛔ 需要关注"
  print "诊断完成: 通过 P 个, 异常 F 个"
```

**关键复用**：doctor 不造新的校验逻辑，直接调 `PluginLoader` 里加载链路用过的两个方法——这样开发时的诊断结果与实际加载判定完全一致，避免"doctor 说 OK，但实际加载又失败"的错位。

## §9 版本范围：`VersionRange` 的匹配规则

`io.leavesfly.jimi.plugin.util.VersionRange` 是插件系统的**版本语言**——`plugin.yaml` 里所有的版本约束（`compatibility.jimi_version` / `compatibility.java_version` / `dependencies[].version`）都经它解析。它是一个 `final` 工具类，只暴露一个静态方法 `matches(range, actualVersion)`。

### 9.1 支持的语法

| 语法 | 含义 | 示例 |
|---|---|---|
| `"1.2.3"` | 等值匹配（无运算符时的默认语义） | `"1.2.3"` |
| `"=1.2.3"` | 显式等于 | `"=1.0.0"` |
| `">=1.0.0"` | 大于等于 | `">=0.1.0"` |
| `"<=2.0.0"` | 小于等于 | `"<=1.9.9"` |
| `">1.0.0"` | 严格大于 | `">0.0.9"` |
| `"<2.0.0"` | 严格小于 | `"<1.0.0"` |
| `">=1.0.0,<2.0.0"` | **AND 组合**（逗号分隔多个约束） | `">=0.1.0,<1.0.0"` |

**不支持**的语法（刻意简化）：

- `^1.0.0` / `~1.2.0` 等 npm 风格缩写
- OR 语义（`||`）
- 预发布标签比较（`1.0.0-alpha < 1.0.0-beta`）

有需要时再扩展——当前取的是"**能覆盖 Jimi 内部 95% 场景的最小子集**"。

### 9.2 版本号解析规则

`VersionRange.parseVersion(String)` 把字符串转成 `int[3]` 数组（`major.minor.patch`），规则：

- 仅保留第一个 `-` 或 `+` 之前的部分，丢弃 `-SNAPSHOT` / `-rc1` / `+build.1` 等后缀
- 段数不足三段时补 `0`：`"17"` → `[17, 0, 0]`；`"1.2"` → `[1, 2, 0]`
- 任一段非数字（解析失败）→ 整个表达式返回 `null`，`matches()` 直接返回 `false`

### 9.3 `matches` 的判定流程

```text
matches(range, actual):
  if range is null or blank:  return true     # 无限制 = 放行
  if actual is null or blank: return false
  actualVer = parseVersion(actual)
  if actualVer is null: return false

  for clause in range.split(","):             # AND 语义
      clause = trim(clause)
      if clause is empty: continue
      if !matchesClause(clause, actualVer):
          return false                        # 任一 clause 失败即整体失败
  return true
```

`matchesClause` 提取前缀运算符（`>=` / `<=` / `>` / `<` / `=`；无前缀则默认 `=`），解析期望版本后三段整数逐位 `Integer.compare` 比较，按运算符判定是否通过。

### 9.4 典型示例与踩坑

| 实际版本 | range 表达式 | 结果 | 说明 |
|---|---|---|---|
| `"0.1.0"` | `">=0.0.1"` | ✅ | |
| `"0.1.0"` | `">=1.0.0,<2.0.0"` | ❌ | 违反 `>=1.0.0` |
| `"17"` | `">=17"` | ✅ | `"17"` 被补成 `[17,0,0]` |
| `"0.1.0-SNAPSHOT"` | `">=0.1.0"` | ✅ | `-SNAPSHOT` 被丢弃，按 `0.1.0` 判定 |
| `"0.1.0-SNAPSHOT"` | `">0.1.0"` | ❌ | `0.1.0 > 0.1.0` 为假 |
| `"1.0.0"` | `"1.0"` | ✅ | range 被补成 `[1,0,0]` |

> **踩坑提醒**：插件作者在 `compatibility.jimi_version` 上写 `">0.1.0"` 会把 `0.1.0-SNAPSHOT` 开发版**排除**——因为 `-SNAPSHOT` 会被丢弃后按等于 `0.1.0` 判定。正确写法是 `">=0.1.0"`。

## §10 兼容性与 Requirements 校验

`PluginLoader.evaluateChecks(PluginSpec)` 在解析完 `plugin.yaml` 后立即执行两道校验，任何一道失败都会把插件标记为 `rejected`：

```text
evaluateChecks(spec):
  r1 = checkCompatibility(spec, getCurrentJimiVersion())
  if !r1.passed(): return LoadedPlugin.rejected(spec, r1.reason())
  r2 = checkRequirements(spec)
  if !r2.passed(): return LoadedPlugin.rejected(spec, r2.reason())
  return LoadedPlugin.accepted(spec)
```

`CheckResult` 是 `PluginLoader` 内的 `record (boolean passed, String reason)`，`ok()` / `fail(reason)` 两个工厂。

### 10.1 `checkCompatibility`：三维度门禁

```text
checkCompatibility(spec, jimiVersion):
  c = spec.getCompatibility()
  if c is null: return ok()                    # 未声明 = 全放行

  # ① Jimi 版本
  if c.jimiVersion not blank and
     !VersionRange.matches(c.jimiVersion, jimiVersion):
      return fail("requires Jimi <range>, but running <actual>")

  # ② Java 版本（读 System.getProperty("java.specification.version")）
  if c.javaVersion not blank and
     !VersionRange.matches(c.javaVersion, currentJava):
      return fail("requires Java <range>, but running <actual>")

  # ③ 操作系统
  if c.os not empty:
      currentOs = detectOsName()               # linux / mac / windows
      if currentOs not in c.os (忽略大小写):
          return fail("requires OS <list>, but running <actual>")

  return ok()
```

**`getCurrentJimiVersion()` 的回退策略**：

```text
getCurrentJimiVersion():
  version = PluginLoader.class.getPackage().getImplementationVersion()
  return version if not blank else DEFAULT_JIMI_VERSION    # "0.1.0"
```

从本类所在 JAR 的 `MANIFEST.MF` 的 `Implementation-Version` 读取，读不到时回退到 `"0.1.0"`，与 `WelcomeRenderer.getVersionInfo` 的策略对齐。**IDE 开发环境**下 JAR manifest 不存在时，插件能看到的 Jimi 版本永远是 `0.1.0`，所以本地开发时 `compatibility.jimi_version: ">=0.1.0"` 是最安全的写法。

**`detectOsName()` 的归一**：把 `System.getProperty("os.name")` 映射到三个固定值：

- `mac` / `darwin` → `"mac"`
- `win` → `"windows"`
- `nux` / `nix` / `aix` → `"linux"`
- 其他 → 原始小写字符串

清单里写 `os: [linux, mac]` 即可同时支持 Linux 与 macOS。

### 10.2 `checkRequirements`：三类环境检查

```text
checkRequirements(spec):
  req = spec.getRequirements()
  if req is null: return ok()

  # ① 可执行命令（遇失败即返回，不继续检后续）
  for bin in req.binaries:
      r = checkBinary(bin)
      if !r.passed(): return r

  # ② 环境变量
  for var in req.envVars:
      value = System.getenv(var)
      if value is null or empty:
          return fail("required env var not set: " + var)

  # ③ 必需文件
  for f in req.files:
      if !resolveRequiredFile(f, spec.getPluginDir()):
          return fail("required file not found: " + f)

  return ok()
```

### 10.3 `checkBinary` 的跨平台策略

`checkBinary(BinaryRequirement)` 是三类检查里最复杂的——它要**真的执行一个子进程**来判定命令是否可用：

| 平台 | Shell | 有 `check` 字段 | 无 `check` 字段（默认探测） |
|---|---|---|---|
| Windows | `cmd.exe /c <cmd>` | 执行 `bin.check` 原样 | `where <safeName> >nul 2>&1` |
| 其他（Linux / macOS / POSIX） | `/bin/sh -c <cmd>` | 执行 `bin.check` 原样 | `command -v '<safeName>' >/dev/null 2>&1` |

**执行规则**：

- **超时**：`REQUIREMENT_CHECK_TIMEOUT_SECONDS = 10L` 秒；超时 → `p.destroyForcibly()` + `fail("check command timed out")`
- **退出码**：`0` 视为通过；非 `0` → `fail("check command exited with code N")`
- **启动失败**（`IOException`）/ **被中断**（`InterruptedException`，会重置中断位）→ `fail("check command failed: " + msg)`
- **错误流合并**：`redirectErrorStream(true)` 把 stderr 并入 stdout，避免 stderr 缓冲区溢出阻塞子进程

**命令注入防护**：`defaultBinaryProbe` 对命令名做最小转义：

- Windows：剔除 `& | < > ^ "` 与空白字符（合法可执行名不会包含这些）
- POSIX：单引号中的 `'` 转义为 `'\''`（标准 POSIX 单引号转义）

**失败消息格式化**：`buildBinaryFailureMessage` 会在失败原因后追加 `install_hint`（如果声明了）：

```
binary 'google-java-format' unavailable (check command exited with code 127); install hint: brew install google-java-format
```

这让用户直接看到安装命令，不用查文档。

### 10.4 `resolveRequiredFile` 的路径解析

```text
resolveRequiredFile(fileRef, pluginDir):
  p = Paths.get(fileRef)
  if p.isAbsolute():
      return Files.exists(p)                   # 绝对路径直查
  if pluginDir != null:
      underPlugin = pluginDir.resolve(fileRef)
      if Files.exists(underPlugin): return true   # 相对插件目录优先
  underCwd = Paths.get(user.dir).resolve(fileRef)
  return Files.exists(underCwd)                # 回退到进程工作目录
```

**典型用法**：
- `files: [pom.xml]` —— 检查项目根目录是否有 `pom.xml`（回退到 `user.dir`）
- `files: [templates/project.tmpl]` —— 检查插件自带模板是否完整（相对 `pluginDir`）

## §11 常见踩坑与设计边界

### 11.1 作者侧：插件开发的 9 个高频踩坑

| # | 现象 | 原因 | 解决 |
|---|---|---|---|
| 1 | 插件被标记为 `⛔ requires Jimi >=1.0.0, but running 0.1.0` | 本地 IDE 跑 Jimi 读不到 MANIFEST，版本永远是 `0.1.0` | `compatibility.jimi_version` 写 `">=0.1.0"` 或去掉 `compatibility` |
| 2 | 插件装了但 `/plugin list` 不显示 | `plugin.yaml` 缺必需字段（`name`/`version`/`description`）→ `validate()` 抛异常被丢弃 | 查日志 `ERROR Failed to load plugin at ...` |
| 3 | `SKILL.md` 放进去了但没注册 | `provides.skills` 白名单不含此 skill，或 `SKILL.md` 里的 `name` 与白名单不匹配 | 对齐 `provides.skills` 与 `SKILL.md#name`；或清空 `provides.skills` 放行全部 |
| 4 | `mcp/servers.json` 写了但 MCP 工具不可用 | MCP 是 **discovery-only**，在会话**创建后**才能被新增的插件影响 | 装插件后 `/plugin reload` 不够，需重启会话让 `JimiFactory` 重新读 `getAllDiscoveredMcpConfigFiles()` |
| 5 | `agents/xxx/agent.yaml` 没生效 | Agents 是 **discovery-only（MVP）**，`AgentModuleAdapter` 不注入 `AgentRegistry` | 当前版本只能放到 `~/.jimi/agents/` 或项目级 `.jimi/agents/` |
| 6 | `/plugin install owner/repo` 404 | 默认分支既不是 `main` 也不是 `master`（如 `trunk` / `dev`） | 显式写 `owner/repo#<branch>` |
| 7 | 单仓库多插件装不上 | 插件根不在仓库顶层 | 用 `owner/repo/<subpath>#<branch>` 指定子路径 |
| 8 | `dependencies` 声明了，依赖插件没自动装 | **MVP 阶段不做传递加载**——`dependencies` 仅做版本范围校验 | 用户需手动先装依赖插件 |
| 9 | `lifecycle.on_install` 脚本没执行 | **MVP 阶段 `PluginLifecycle` 只存字段，执行器未实装** | 待后续版本；当前可改为自定义 Hook |

### 11.2 用户侧：运行时的 4 个踩坑

| # | 现象 | 原因 | 解决 |
|---|---|---|---|
| 1 | `/plugin uninstall` 说找不到，但 `/plugin list` 里明明有 | 该插件是 CLASSPATH 或 PROJECT 层的，`uninstall` 只删 USER 层 | 手动删项目的 `.jimi/plugins/<name>/` 或从 classpath 移除 |
| 2 | `/plugin reload` 后某个插件"没了" | 实际上是从磁盘重扫——插件文件被外部删除/改名了，`reload` 如实反映磁盘状态 | 查 `~/.jimi/plugins/` 和 `<project>/.jimi/plugins/` 是否在 |
| 3 | `/plugin disable <name>` 后 Skill 依旧出现在 System Prompt 的 `<available_skills>` 摘要里 | `SkillRegistry.generateSkillsSummary()` 由 `JimiFactory` 在**会话创建时**调用一次并写入 System Prompt，`SkillRegistry.unregister()` 只改内存索引、不回写当前会话的 Prompt | 新开一个会话；Prompt 会按新的 Skill 注册表重新生成 |
| 4 | IDE 直接 Run 时 `hello-world` 插件没出现 | `PluginLoader.TEST_FIXTURE_PLUGIN_NAMES` 主动过滤了该名，避免测试夹具污染运行时 | 改名即可；或挪到 `~/.jimi/plugins/<new-name>/` 走 USER 层 |

### 11.3 当前设计的边界清单

**MVP 阶段刻意未做**（源码注释里标注为"待实装"/"Phase 2"/"后续阶段"）：

- `PluginLifecycle` 的脚本执行器（4 个生命周期脚本字段只存不执行）
- `PluginDependency` 的传递加载（仅做版本范围校验，不会自动拉取依赖插件）
- `McpModuleAdapter` 的运行时增删（`MCPToolProvider.mcpConfigFiles` 是会话级不可变）
- `AgentModuleAdapter` 的真实注入（`AgentSpecLoader` 用 `@PostConstruct` 写死扫描路径）
- 内置插件白名单（`BUILTIN_CLASSPATH_PLUGINS = {}`，目前没有官方内置插件）

**刻意做的**（非 bug，是设计选择）：

- `PluginRegistry` 与三个原生 Registry 用 `@DependsOn` 绑定启动顺序——保证插件追加注册时原生数据已就位
- `/plugin install` 的 GitHub 404 fallback 只在**未显式指定分支**时生效，显式分支的 404 直接抛错——避免把用户的 typo 掩盖成"分支不存在"
- `PluginInstaller.copyToUserPlugins` 三阶段原子切换——升级失败不会把旧版本带下水
- `ArchiveInstaller.unzipFile` 的 Zip Slip 防护——即便恶意 ZIP 也无法写出到解压目录之外

### 11.4 扩展边界：如何新增一类扩展点

当你想把插件系统扩展到一个**新的**扩展点（比如"自定义 LLM Provider 插件"），需要：

1. 在 `PluginProvides` 里加一个白名单字段（如 `llmProviders`）+ `allowsLlmProvider(name)`
2. 在 `PluginModuleToggle` 里加对应开关（`llmProviders: true` 默认）
3. 在 `PluginModuleToggle.isModuleEnabled` 的 switch 里添加新 case
4. 实现一个 `LlmProviderModuleAdapter implements PluginModuleAdapter`（`@Component`），`getModuleName()` 返回 `"llm_providers"`，在 `load` 里扫子目录 + 查白名单 + 调 `LLMFactory` 注册
5. 不需要改 `PluginDispatcher`——Spring 自动注入

这是**零侵入扩展**设计的最大回报：核心调度器对扩展点数量完全不感知。

## §12 小结

Jimi 插件系统用**一个 `plugin.yaml`** 把 5 种扩展点打包成了一个有版本号、可分发、可审计、可灰度的产品单元，同时在架构上做了 3 件关键的事：

1. **零侵入桥接**：通过 `PluginModuleAdapter` SPI 把原生 Registry 与插件包解耦，插件系统整体可拔插
2. **三层作用域与磁盘重扫 reload**：与 Skills/Hooks/Commands 的三层语义对齐，`reload` 真 · 从磁盘读，避免"幽灵插件"
3. **原子切换安装 + Zip Slip 防护**：让"升级中途失败不损坏旧版"和"恶意归档无法逃出解压目录"成为底层保障
