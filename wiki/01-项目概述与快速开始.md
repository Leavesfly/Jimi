# 01 · 项目概述与快速开始

> 用一篇文档理解 Jimi 是什么、能做什么、如何跑起来。

---

## 1. 项目定位

**Jimi** 是一个纯 Java 实现、面向 Java 开发者的**开源 AI CLI Agent**，目标是提供类 Claude Code 的终端式结对编程体验，同时保留 Java 生态的工程化特性（Spring Boot、响应式、可嵌入）。

它不是"另一个 ChatGPT 客户端"，而是一个**以 LLM 为推理引擎、以工具调用为行动能力、以 ReAct 循环为调度骨架**的通用智能代理框架。在 Jimi 中：

- **LLM** 负责"思考"（推理 / 规划 / 总结）
- **Tools** 负责"行动"（读写文件 / 执行命令 / 查图谱 / 调外部服务）
- **Agent** 定义"角色"（提示词 + 可用工具集合）
- **Engine** 负责"编排"（把上述三者串起来跑起来）

### 一句话总结

> Jimi = **ReAct Loop + 工具生态 + 多 Agent + 知识增强** 的 Java 版 Claude Code。

## 2. 核心能力

### 2.1 六大能力支柱

| 支柱 | 关键模块 | 解决什么问题 |
|------|----------|--------------|
| **多 Agent 协作** | `AgentRegistry` / `SubAgentTool` / `TeamManager` | 让不同角色（架构师 / 编码 / 评审 / DevOps）各司其职 |
| **丰富工具生态** | `ToolRegistry` + 18 个内置工具 + MCP 动态工具 | LLM 不再只会"说"，还能读写文件、跑命令、调服务 |
| **Skills 技能包** | `SkillRegistry` / `SkillsTool` | 把领域知识（API 设计、Java 最佳实践）模块化 |
| **Hooks 自动化** | `HookRegistry` / 15 种事件 | 工作流自动化（保存后格式化、提交前测试） |
| **代码图谱 + RAG** | `GraphManager` / `RagManager` | 基于 AST 的代码理解，语义 + 结构混合检索 |
| **记忆与上下文** | `MemoryManager` / `SimpleCompaction` | ReCAP 式长程记忆与 Token 优化 |

> 18 个内置工具对应：文件类 5 个（`ReadFile`/`WriteFile`/`StrReplaceFile`/`Grep`/`Glob`）、Shell 1 个（`BashTool`）、Web 2 个（`FetchURL`/`WebSearch`）、图谱 3 个（`CodeLocateTool`/`CallGraphTool`/`ImpactAnalysisTool`）、任务 2 个（`SubAgentTool`/`TeamAgentTool`）、交互 1 个（`AskHuman`）、知识 3 个（`MemoryTool`/`SkillsTool`/`SetTodoList`）、元工具 1 个（`MetaTool`），再加 `MCPTool` wrapper 动态挂载任意 MCP 服务端工具。

### 2.2 工程特性

- **响应式全链路**：Reactor `Mono` / `Flux` 贯穿 LLM 调用、工具执行、Wire 事件总线
- **Spring Boot 驱动**：依赖注入、自动装配、配置外部化
- **可嵌入**：支持 `jimi.embedded=true` 模式，作为库嵌入到其他 Java 应用（如 IntelliJ 插件）
- **会话可持久化**：JSONL 格式历史文件，支持 `--continue` 断点恢复
- **审批与 YOLO**：敏感操作默认人工审批，`-y` 一键放行所有操作

## 4. 快速开始

### 4.1 环境要求

- **JDK 17+**（建议 17 或 21 LTS）
- **Maven 3.6+**
- **操作系统**：macOS / Linux / Windows（推荐 Unix 类系统以便使用 Shell 工具）
- **LLM API Key**（至少一个）：OpenAI / Moonshot / 通义千问 / DeepSeek / Ollama 等

### 4.2 三分钟上手

```bash
# 1. 克隆项目
git clone https://github.com/leavesfly/Jimi.git
cd Jimi

# 2. 一键安装（编译 + 初始化配置 + 环境检查）
./scripts/quick-install.sh

# 3. 启动（默认 Agent）
./scripts/start.sh
```

启动后进入交互式 Shell，输入自然语言即可开始：

```
> 帮我分析这个项目的入口类，并给出优化建议
```

### 4.3 分步安装

如果希望对每一步可控：

```bash
./scripts/check-env.sh       # 检查 JDK / Maven 是否满足要求
./scripts/build.sh           # mvn clean package
./scripts/init-config.sh     # 在 ~/.jimi/ 下初始化配置
./scripts/start.sh           # 启动
```

### 4.4 配置 LLM

Jimi 的真实配置加载路径定义在 `JimiConfiguration.ConfigLoaderHelper`：

- 用户配置文件：**`~/.jimi/config.yml`**（YAML 格式，注意后缀是 `.yml`）
- 如用户目录下文件不存在，回退到 classpath 里的**内置默认配置** `src/main/resources/.jimi/config.yml`

> ⚠️ 仓库中的 `scripts/init-config.sh` 会向导式生成一份 `~/.jimi/config.json`，但**当前版本的 `JimiConfig` 只加载 `.yml`**。如果你是第一次用，建议直接把 `src/main/resources/.jimi/config.yml` 拷贝到 `~/.jimi/config.yml` 后按需改 `api_key`。

一份最小可用的配置示例（字段名与 `JimiConfig` / `LLMProviderConfig` / `LLMModelConfig` 的 `@JsonProperty` 完全一致）：

```yaml
# 默认模型（必须在 models 中已定义）
default_model: qwen3-max

providers:
  qwen:
    type: qwen                                                   # ProviderType 枚举之一
    base_url: https://dashscope.aliyuncs.com/compatible-mode/v1
    api_key: sk-xxxxxxxxxxxxxxxx
    custom_headers: {}
    rate_limit:                                                  # 可选
      window_ms: 4000
      max_requests: 4
      sleep_ms: 3000

models:
  qwen3-max:
    provider: qwen                                               # 必须对应 providers 的某个 key
    model: qwen3-max                                             # 发给 LLM API 的真实模型名
    max_context_size: 200000                                     # 上下文窗口（Token）
```

`LLMProviderConfig.ProviderType` 当前支持的枚举值：`kimi` / `deepseek` / `qwen` / `ollama` / `openai` / `claude` / `cursor` / `glm` / `minimax`。

支持的 Provider 种类与 ChatProvider 类映射关系详见 [05 · LLM 接入层与多模型支持](05-LLM接入层与多模型支持.md)。

### 4.5 常见启动参数

`CliApplication` 使用 Picocli 定义了下列选项：

| 参数 | 说明 |
|------|------|
| `-w, --work-dir <path>` | 指定工作目录（默认 `System.getProperty("user.dir")`） |
| `-m, --model <name>` | 覆盖使用的模型（优先级：此参数 > Agent 配置的 `model` > 全局 `default_model`） |
| `-C, --continue` | 继续上次会话（`SessionManager` 从 `~/.jimi/metadata.json` 按工作目录查 `lastSessionId`） |
| `-y, --yolo, --yes` | YOLO 模式：`Approval` 自动批准所有工具调用（慎用） |
| `--agent-file <path>` | 指定自定义 Agent 规范文件（YAML），不指定则加载默认 Agent |
| `--mcp-config-file <path>` | 额外加载 MCP 服务配置（可重复指定多个文件） |
| `-c, --command <text>` | 非交互式：执行单条命令后退出（脚本友好，跳过 ShellUI） |
| `--verbose` | 打印详细的启动信息（`JimiConfig.toString`） |
| `--debug` | 启用 `DebugLogger`，输出 LLM 请求/响应、工具调用等详细日志 |

源码位置：`io.leavesfly.jimi.CliApplication`（使用 Picocli 的 `@Command` / `@Option` 注解）。

## 6. 5 分钟上手：一次完整的对话流

假设用户输入：`> 读取 pom.xml 并告诉我依赖了哪些核心库`

Jimi 内部发生的事情（简化版）：

```
用户输入
   │
   ▼
┌──────────────────────────────────────────┐
│ 1. ShellUI 捕获输入，经 Wire 发给 Engine   │
├──────────────────────────────────────────┤
│ 2. JimiEngine.run() → AgentExecutor       │
│    · 注入用户消息到 Context               │
│    · 触发 USER_PROMPT_SUBMIT Hook         │
│    · 按需拉取相关 Topic 记忆              │
├──────────────────────────────────────────┤
│ 3. ReactLoop 第 1 步                     │
│    · 调 LLM.generateStream(sysPrompt,    │
│      history, toolSchemas)                │
│    · LLM 决定调用 ReadFile 工具            │
├──────────────────────────────────────────┤
│ 4. ToolDispatcher 执行 ReadFile           │
│    · Approval 审批（YOLO 跳过）           │
│    · 触发 PRE_TOOL_USE / POST_TOOL_USE    │
│    · 返回文件内容到 Context                │
├──────────────────────────────────────────┤
│ 5. ReactLoop 第 2 步                     │
│    · LLM 读到工具结果，生成总结文本         │
│    · 无工具调用 → 循环结束                 │
├──────────────────────────────────────────┤
│ 6. Context 自动 checkpoint 到 JSONL       │
│    触发 STOP Hook，Wire 推流到 UI         │
└──────────────────────────────────────────┘
```

完整时序与源码分析见 [02 · 系统架构与核心引擎](02-系统架构与核心引擎.md)。
