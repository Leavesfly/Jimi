# 第 12 篇：扩展开发指南

> 本篇是 **Jimi 系列 Wiki** 的收官之作，目标是给**二次开发者**一份可动手的实操手册。前 11 篇自上而下拆解了 Jimi 的内部结构，本篇自下而上回答一个问题：**"我要给 Jimi 增加 X，该怎么做？"**
>
> 本篇覆盖 6 大扩展点：
> 1. **自定义 Tool** —— 继承 `AbstractTool`，成为 LLM 可调用的原生工具
> 2. **Skill 技能包** —— Markdown + YAML Front Matter，按需注入 system prompt
> 3. **Hook 钩子** —— YAML 声明式，在 15 个生命周期事件上挂载脚本
> 4. **自定义 Slash 命令** —— YAML 声明式，扩展 `/xxx` 命令
> 5. **LLM ChatProvider** —— 接入新的大模型厂商
> 6. **ToolProvider SPI** —— 批量注入一组工具（如 MCP Server、插件市场）
>
> 每一节的结构固定为：**适用场景 → 最小模板 → 契约速览 → 注册方式 → 验证方法 → 踩坑提示**。所有代码片段均来源于真实仓库的接口签名（不臆造）。

## §1 扩展点全景

### 1.1 六大扩展点对比

| 扩展点 | 契约形态 | 加载时机 | 生效范围 | 典型用途 | 所需技能 |
|---|---|---|---|---|---|
| **自定义 Tool** | Java 接口 `Tool<P>` | `ToolRegistryFactory.create` | 单个 Agent（由 AgentSpec 决定） | 给 LLM 新增一个可调用的原生工具 | Java + Spring |
| **Skill 技能包** | Markdown + YAML Front Matter | `SkillLoader` 扫描目录 | 所有 Agent 共享（通过 `Skill` 工具按需激活） | 注入领域知识、最佳实践到 system prompt | Markdown |
| **Hook 钩子** | YAML 声明式 | `HookLoader` 扫描目录 | 会话级全局（按 `HookType` 触发） | 在工具调用前/后执行脚本（格式化、审计、审批） | YAML + Shell |
| **自定义 Slash 命令** | YAML 声明式 <br> **或** Java `CommandHandler` | `CustomCommandLoader` / Spring | CLI 交互式会话 | 定义 `/xxx` 命令快捷键 | YAML 或 Java |
| **LLM ChatProvider** | Java 接口 `ChatProvider` | `LLMFactory.createLLM` | 全局（按模型名切换） | 接入新大模型厂商 | Java + HTTP/流式 |
| **ToolProvider SPI** | Java 接口 `ToolProvider` | Spring 注入 + `ToolRegistryFactory` 按 `getOrder()` 分发 | 所有 Agent（满足 `supports()` 的） | 批量注入一组工具（如 MCP、插件市场） | Java + Spring |

### 1.2 选型决策树

```
我要扩展什么？
├── 只是给 LLM 多一个工具能力？
│   ├── 单个工具，逻辑简单/中等 → §2 自定义 Tool
│   └── 一组工具，有统一来源/生命周期 → §7 ToolProvider SPI
│
├── 想改变 LLM 的"说话方式/知识范围"？
│   ├── 只加静态知识/风格指令 → §3 Skill 技能包（无代码，纯 Markdown）
│   └── 要在工具调用前后做动作（格式化/审批/通知）→ §4 Hook 钩子
│
├── 想在 CLI 交互中加快捷指令？
│   ├── 只是固定 prompt 模板 → §5.2 YAML 声明式命令
│   └── 需要 Java 逻辑（查状态、改会话）→ §5.3 Java 编程式命令
│
├── 想接入新的大模型？
│   └── §6 LLM ChatProvider
│
└── 想接入外部工具市场（Model Context Protocol 规范）？
    └── 配置 MCP Server（不改 Java 代码，见 11 篇 §6.2）
```

### 1.3 "配置还是代码"的权衡

| 选择"配置"（Skill / Hook / YAML 命令 / MCP） | 选择"代码"（Tool / ToolProvider / ChatProvider / Java 命令） |
|---|---|
| ✅ 非技术人员可写 | ✅ 精确控制参数 schema、返回类型 |
| ✅ 热更新无需重新编译 | ✅ 可访问 Spring 上下文、`JimiRuntime`、`Session` |
| ✅ 配置即文档 | ✅ 可做复杂校验、异步流程 |
| ❌ 逻辑能力受限（Skill 纯文本、Hook 只能调脚本） | ❌ 需重新打包发布 |
| ❌ 热路径性能稍差（YAML 解析、进程 fork） | ❌ 需读 Java 代码才能理解行为 |

**经验法则**：**先用配置原型验证，后用代码稳定化**。例如一个新工具，可以先用 MCP Server（Python 快速实现）验证 LLM 能否正确调用，再用 Java Tool 改写成内置工具。

## §2 自定义 Tool（继承 AbstractTool）

### 2.1 适用场景

- 给 LLM 增加**一个**新的可调用能力（查数据库、调内部 API、读特殊格式文件）
- 需要精确控制参数 schema（用 Java 类 + `@JsonPropertyDescription` 注解定义）
- 需要访问 Spring 容器里的其他 Bean（如 `MemoryManager`、`Approval`、`SandboxValidator`）
- 工具执行可能耗时，希望走 Reactor 的 `Mono.defer` 异步链路
- 需要被 `Approval` 审批机制拦截（如执行 shell、修改文件等副作用操作）

**不适合的场景**：
- 一次要加 10+ 个工具且共享生命周期 → 用 **§7 ToolProvider SPI**
- 只是想注入一段静态指令到 system prompt → 用 **§3 Skill 技能包**
- 工具来自外部进程且遵循 MCP 协议 → 直接配置 MCP Server（见 11 篇）

### 2.2 最小模板

参考 `BashTool.java`（`tool/core/BashTool.java`）的真实结构。下面是一个"查询 Git 最近提交"的示例工具：

```java
package io.leavesfly.jimi.tool.core;

import com.fasterxml.jackson.annotation.JsonPropertyDescription;
import io.leavesfly.jimi.tool.AbstractTool;
import io.leavesfly.jimi.tool.ToolResult;
import lombok.*;
import org.springframework.beans.factory.config.ConfigurableBeanFactory;
import org.springframework.context.annotation.Scope;
import org.springframework.stereotype.Component;
import reactor.core.publisher.Mono;

/**
 * GitRecentCommits 工具：查询当前工作目录 Git 仓库的最近提交。
 */
@Component
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)   // ⚠️ 必须是 prototype
public class GitRecentCommitsTool extends AbstractTool<GitRecentCommitsTool.Params> {

    /** 参数 POJO，必须是 public static class */
    @Data
    @Builder
    @NoArgsConstructor
    @AllArgsConstructor
    public static class Params {
        @JsonPropertyDescription("返回的提交条数，1~50，默认 10")
        @Builder.Default
        private int count = 10;

        @JsonPropertyDescription("可选：仅返回指定作者的提交")
        private String author;
    }

    public GitRecentCommitsTool() {
        super(
            "GitRecentCommits",                               // 工具名，LLM 看到的 name
            "查询当前 Git 仓库的最近提交记录，可按作者过滤。",   // 工具描述，LLM 据此决定是否调用
            Params.class                                       // 参数类型，用于反射生成 JSON Schema
        );
    }

    @Override
    public boolean isConcurrentSafe() {
        return true;   // 只读工具，可并发
    }

    @Override
    public Mono<ToolResult> execute(Params params) {
        return Mono.defer(() -> {
            if (params.count < 1 || params.count > 50) {
                return Mono.just(ToolResult.error(
                    "count 必须在 1~50 之间，当前值=" + params.count,
                    "参数错误"));
            }
            try {
                // ... 实际调用 git log 的逻辑 ...
                String output = "abc1234 feat: add xxx\n...";
                return Mono.just(ToolResult.ok(
                    output,                                    // 完整输出（给 LLM）
                    "查询到 " + params.count + " 条提交",        // 人类可读 message
                    "Git 提交 " + params.count + " 条"));        // 简要 UI 显示
            } catch (Exception e) {
                return Mono.just(ToolResult.error(
                    e.getMessage(),
                    "查询 Git 提交失败"));
            }
        });
    }
}
```

### 2.3 契约速览

**`Tool<P>` 接口**（源码 `tool/Tool.java`）：

| 方法 | 必需 | 说明 |
|---|---|---|
| `String getName()` | ✅ | 工具名（LLM 看到的 name，必须唯一；由 `AbstractTool` 构造器传入） |
| `String getDescription()` | ✅ | 工具描述（LLM 据此决策；由 `AbstractTool` 构造器传入） |
| `Class<P> getParamsType()` | ✅ | 参数类型（用于反射生成 JSON Schema；由 `AbstractTool` 构造器传入） |
| `Mono<ToolResult> execute(P params)` | ✅ | **唯一需要自己实现的方法**；必须返回 `Mono`，不能阻塞 |
| `boolean validateParams(P params)` | default `true` | 可选参数校验钩子 |
| `boolean isConcurrentSafe()` | default `true` | ⚠️ **写操作必须覆盖为 `false`**（否则多个 LLM 调用会并发改同一文件） |
| `JsonNode getCustomParametersSchema()` | default `null` | 返回非 null 会跳过反射生成，直接使用自定义 schema（MCP 工具就用这个机制） |

**`AbstractTool<P>` 基类**（源码 `tool/AbstractTool.java` 仅 36 行）：只提供三个 final 字段（name/description/paramsType）+ 三个 getter，**没有任何其他模板方法**。所有逻辑靠你自己在 `execute` 里实现。

**`ToolResult` 工厂方法**（源码 `tool/ToolResult.java`）：

| 工厂 | 签名 | 用途 |
|---|---|---|
| `ToolResult.ok(output, message)` | 2 参 | 成功，`brief` 为空 |
| `ToolResult.ok(output, message, brief)` | 3 参 | 成功，附 UI 简要 |
| `ToolResult.error(message, brief)` | 2 参 | 错误，`output` 为空 |
| `ToolResult.error(output, message, brief)` | 3 参 | 错误，附详细输出（如 shell 命令的 stderr） |
| `ToolResult.rejected()` | 0 参 | 被用户拒绝（Approval 机制内部用，业务代码一般不手动调） |

**参数注解**：
- `@JsonPropertyDescription("...")` —— 会反射成 JSON Schema 的 `description`，LLM 能看到
- `@Builder.Default` —— 默认值（否则 Lombok 不会生成默认值）
- 字段必须是 Java Bean 风格（`@Data` 提供 getter/setter）

### 2.4 注册方式

**推荐方式 1：加到 `createStandardRegistry`（内置工具）**

编辑 `tool/ToolRegistryFactory.java#createStandardRegistry` 方法，以 diff 展示需要追加的两行：

```diff
  public ToolRegistry createStandardRegistry(...) {
      ToolRegistry registry = new ToolRegistry();
      // ... 已有的工具注册（BashTool、ReadFileTool 等） ...
+
+     // 新增：注册自定义的 GitRecentCommitsTool
+     GitRecentCommitsTool gitTool =
+         applicationContext.getBean(GitRecentCommitsTool.class);
+     registry.register(gitTool);

      return registry;
  }
```

这样所有 Agent 默认都能用。

**推荐方式 2：通过 AgentSpec 选择性暴露**

如果只想让某些 Agent 用到这个工具，改走 **§7 ToolProvider SPI**——在 `supports()` 里根据 `agentSpec.getName()` 过滤。

**关键约束**：
- 工具类必须标注 `@Component` + `@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)`
  - 前者让 Spring 扫描到
  - 后者让每次 `getBean` 返回**新实例**（避免会话间状态污染——`BashTool` 就是这样做的）
- 必须在 `io.leavesfly.jimi` 包下或子包，否则 Spring 扫描不到（见主启动类的 `@SpringBootApplication` 默认扫描范围）

### 2.5 验证方法

1. **单元测试**：不需要启动 Spring，直接 `new GitRecentCommitsTool()` 然后 `.execute(params).block()` 即可。参考 `MockChatProvider`（测试工具）的写法。
2. **schema 检查**：启动 Jimi 后，看启动日志里 `Registered N tools` 包含你的工具；也可以写个临时脚本调用 `ToolRegistry.generateToolSchema(tool)` 打印 JSON Schema，确保 LLM 看到的字段描述正确。
3. **LLM 侧验证**：在 Jimi 交互中用自然语言引导（"查一下最近 5 次 git 提交"），观察 LLM 是否能正确调用、参数是否正确解析。
4. **Approval 验证**：如果工具涉及副作用，需接入 `Approval`。看 `BashTool.setApproval` 的注入方式。

### 2.6 踩坑提示

- ❌ **忘记 `@Scope("prototype")`** → 工具会被 Spring 复用为单例，多个会话共享状态，导致诡异的并发 bug。内置所有工具都是 prototype（见 `BashTool`、`MemoryTool` 等）。
- ❌ **在 `execute` 里直接阻塞** → 如 `Thread.sleep`、同步 HTTP 调用、同步文件读。必须用 `Mono.fromCallable(...).subscribeOn(Schedulers.boundedElastic())` 包裹，否则会卡住整个 Reactor 线程池。
- ❌ **参数类用 lambda/匿名类** → `getParamsType()` 反射拿不到字段名，JSON Schema 会退化成空对象。**必须 `public static class` + Lombok `@Data`**。
- ❌ **`getName()` 与其他工具重名** → `ToolRegistry.register` 行为视实现而定（见 04 篇），最坏会被静默覆盖。建议用 PascalCase 命名并加前缀避免碰撞（如 `GitRecentCommits` 而非 `recent`）。
- ❌ **description 写得太随意** → LLM 选工具完全靠 description；写得不清晰 LLM 会不调用或错误调用。推荐格式：**动词开头 + 主要用途 + 限制条件**（如 `BashTool` 的描述）。
- ❌ **写操作没设 `isConcurrentSafe=false`** → LLM 一次返回多个工具调用时会并行执行，两个 `WriteFile` 工具并发写同一文件就会冲突。
- ⚠️ **参数类型如果是 `Map<String, Object>`** → 反射生成的 schema 退化为 `{"type": "object"}`（无字段）。要么改用强类型 POJO，要么覆写 `getCustomParametersSchema()` 手动提供（MCP 工具就走这条路，见 11 篇 §5.1）。

## §3 Skill 技能包

### 3.1 适用场景

- 给 LLM 注入**领域知识**（项目代码规范、业务术语、常用命令速查）
- 定义**最佳实践**（"使用 TDD 时的工作流"、"线上发布 checklist"）
- 写给 LLM 的**指令模板**（"当用户提到 xxx 时，先做 yyy 再做 zzz"）
- 希望**非 Java 开发者**也能贡献内容（产品、DBA、前端同学都能写 Markdown）

**不适合的场景**：
- 需要运行代码/脚本（Skill 是纯文本，没有执行能力）→ 用 **§4 Hook**
- 需要 LLM 调用工具（Skill 不能定义 Tool schema）→ 用 **§2 自定义 Tool**
- 一次性 prompt 模板（不需要常驻注入）→ 用 **§5 Slash 命令**

### 3.2 最小模板

Skill 是一个 `SKILL.md` 文件，**顶部 YAML Front Matter + 下方 Markdown 正文**：

```markdown
---
name: jimi-style-guide
description: Jimi 项目的代码风格规范，包含命名、日志、Reactor 使用等约定
version: 1.0.0
category: development
triggers:
  - 代码风格
  - 编码规范
  - code style
  - lombok
---

# Jimi 代码风格指南

## 1. 命名约定
- 类名使用 PascalCase（如 `BashTool`、`SkillLoader`）
- Spring Bean 使用 `@Component` + `@Scope(SCOPE_PROTOTYPE)`（所有 Tool 都是原型）
- 测试类以 `Test` 结尾（如 `TestKimiChatProvider`）

## 2. Reactor 使用
- 工具的 `execute(params)` 必须返回 `Mono<ToolResult>`
- 阻塞调用用 `.subscribeOn(Schedulers.boundedElastic())` 隔离
- 流式 LLM 响应用 `Flux<ChatCompletionResult>`

## 3. 日志
- 用 `@Slf4j` + Lombok，不手动 `LoggerFactory.getLogger`
- 用 `{}` 占位符，不用字符串拼接（`log.info("loaded {} tools", n)`）

## 4. 错误处理
- 工具失败返回 `ToolResult.error(...)`，不抛异常（否则整个 Reactor 链路断开）
- 配置错误在启动时 fail-fast（见 `LLMFactory.validateConfiguration`）
```

### 3.3 契约速览

**`SkillSpec` 字段**（源码 `skill/SkillSpec.java`）：

| 字段 | 必需 | 类型 | 说明 |
|---|---|---|---|
| `name` | ✅ | String | Skill 唯一标识（建议 kebab-case） |
| `description` | ✅ | String | 简短描述，建议 50 字以内（索引查询时会展示） |
| `content` | ✅ | String | Markdown 正文（自动从 Front Matter 之后的部分提取） |
| `scope` | ✅ | SkillScope | `GLOBAL` / `PROJECT`（由加载目录自动确定） |
| `version` | ⚪ | String | 语义化版本，默认 `1.0.0` |
| `category` | ⚪ | String | 分类标签（development/documentation/testing 等） |
| `triggers` | ⚪ | List\<String\> | 触发关键词，默认 `[]`（用于智能匹配和激活） |
| `dependencies` | ⚪ | List\<String\> | 依赖的其他 Skill 名称，默认 `[]`（组合型 Skill） |

**Front Matter 解析规则**（源码 `SkillLoader.parseSkillContent`）：
- 正则 `^---\s*\n(.*?)\n---\s*\n(.*)$` 匹配
- **缺失 Front Matter 时**会打 warn 日志，整个文件作为 `content`，其他字段走默认值（但 `name`/`description` 必填，后续会校验失败）
- YAML 解析使用 Spring 注入的 `yamlObjectMapper`（Snake-yaml）

### 3.4 注册方式

**加载来源**（优先级从高到低，源码 `SkillLoader.getGlobalSkillsDirectories`）：

| 位置 | Scope | 生效范围 | 适用 |
|---|---|---|---|
| `classpath:/skills/*.md` | GLOBAL | 内置（打进 JAR） | Jimi 官方内置 Skill |
| `~/.jimi/skills/*.md` | GLOBAL | 用户全局 | 跨项目通用 Skill |
| `<project>/.jimi/skills/*.md` | PROJECT | 仅当前项目 | 项目专属规范 |

**加载行为**：
- **JAR 模式下**：跳过 classpath 目录扫描（源码 `isRunningFromJar`），仅从用户目录加载
- **合并加载**：三层都加载，`name` 相同时后加载的**覆盖**先加载的（即 Project > User > Classpath）
- **激活机制**：Skill 不是默认全部注入 system prompt，而是通过 `SkillsTool` 由 LLM 按需激活——参考 06 篇

### 3.5 踩坑提示

- ❌ **Front Matter 两端的 `---` 前后不能有空格** → 正则严格匹配 `^---\s*\n`，`--- ` 会被当成内容开始，整个文件解析失败
- ❌ **YAML 缩进用 Tab** → YAML 禁止 Tab 缩进，`yamlObjectMapper` 会抛错，Skill 加载失败静默跳过（只在 DEBUG 日志里看到）
- ❌ **`name` 包含空格或中文** → 会被 `SkillsTool` 用作查询键，空格/特殊字符会导致激活失败
- ❌ **`content` 写得太长（>10K token）** → 激活时会占用大量 context，挤压对话空间。建议单个 Skill < 2000 字，拆分成多个小 Skill
- ⚠️ **`triggers` 关键词重叠** → 多个 Skill 匹配同一触发词时，激活顺序取决于 `SkillRegistry` 的内部实现，可能非确定性
- ⚠️ **`dependencies` 循环依赖** → 当前 `SkillLoader` 不检测循环（`A → B → A`），会导致激活时栈溢出或重复注入

## §4 Hook 钩子

### 4.1 适用场景

- **工具调用前**：参数校验、权限审批、危险命令拦截（`PRE_TOOL_USE`）
- **工具调用后**：代码格式化、Git 提交、缓存刷新（`POST_TOOL_USE`）
- **工具调用失败时**：告警通知、自动修复、错误日志归档（`POST_TOOL_USE_FAILURE`）
- **会话边界**：会话启动时加载环境变量、会话结束时清理临时文件（`SESSION_START` / `SESSION_END`）
- **用户输入**：敏感词过滤、注入上下文（`USER_PROMPT_SUBMIT`）

**不适合的场景**：
- 扩展 LLM 能力 → Hook 只能调用**外部脚本**，不能给 LLM 加工具；要加工具用 **§2**
- 改变 LLM 回答风格 → 用 **§3 Skill** 或 Agent system prompt

### 4.2 最小模板

配置文件：`<project>/.jimi/hooks/auto-format.yaml` 或 `~/.jimi/hooks/auto-format.yaml`

```yaml
name: auto-format-java
description: 写入 Java 文件后自动用 google-java-format 格式化
enabled: true

trigger:
  type: post_tool_call          # 可用值见 §4.3 HookType
  tools:
    - WriteFile
    - StrReplaceFile
  file_patterns:
    - "*.java"

execution:
  type: script
  script: google-java-format -i "${MODIFIED_FILE}"
  async: true                   # true = 不阻塞工具调用链路
```

**更复杂示例：PRE_TOOL_USE 危险命令拦截**

```yaml
name: block-rm-rf
description: 拦截 rm -rf 命令，避免误删
enabled: true

trigger:
  type: pre_tool_call
  matcher: "Bash"               # 正则匹配工具名（对齐 Claude Code）

execution:
  type: script
  script: |
    # tool_input 不在 ${VAR} 替换清单里，必须从 stdin JSON 解析
    COMMAND=$(jq -r '.tool_input.command // empty' </dev/stdin)
    if echo "$COMMAND" | grep -Eq 'rm\s+-rf\s+/'; then
      echo "禁止执行 rm -rf /" >&2
      exit 2                    # exit=2 = 阻塞操作（对齐 Claude Code）
    fi
```

### 4.3 契约速览

**`HookSpec` 字段**（源码 `core/hook/HookSpec.java`）：

| 字段 | 必需 | 说明 |
|---|---|---|
| `name` | ✅ | Hook 唯一标识 |
| `description` | ✅ | 必需描述（`validate()` 会抛错） |
| `trigger` | ✅ | 触发条件（见下表） |
| `execution` | ✅ | 执行配置（复用 `ExecutionSpec`，与 Custom Command 共享） |
| `enabled` | ⚪ | 默认 `true` |
| `conditions` | ⚪ | 额外条件列表（默认 `[]`） |
| `priority` | ⚪ | 优先级，默认 `0`；数值越大越先执行 |

**`HookType` 枚举**（源码 `core/hook/HookType.java`，共 15 个）：

| 分组 | 类型 | 触发时机 |
|---|---|---|
| **Claude Code 对齐（9 个）** | `USER_PROMPT_SUBMIT` | 用户输入提交时 |
| | `PRE_TOOL_USE` | 工具执行前（可拦截） |
| | `POST_TOOL_USE` | 工具执行成功后 |
| | `POST_TOOL_USE_FAILURE` | 工具执行失败后 |
| | `NOTIFICATION` | 系统通知 |
| | `STOP` | Agent 准备停止响应时 |
| | `SUBAGENT_STOP` | 子 Agent 停止时 |
| | `SESSION_START` | 会话启动 |
| | `SESSION_END` | 会话结束 |
| **Jimi 扩展（3 个）** | `PRE_AGENT_SWITCH` / `POST_AGENT_SWITCH` | Agent 切换前后 |
| | `ON_ERROR` | 系统错误时 |
| **Teams 扩展（3 个）** | `TEAM_START` / `TEAM_COMPLETE` / `TEAMMATE_TASK_COMPLETE` | 团队协作事件 |

**命名兼容**（源码 `HookType.fromString`）：同时支持 `PreToolUse`（Claude Code 风格）、`pre_tool_use`（snake_case）、`pre-tool-use`（kebab-case），以及旧命名 `PRE_TOOL_CALL` 自动映射为 `PRE_TOOL_USE`。

**`HookTrigger` 过滤字段**：

| 字段 | 说明 | 适用 HookType |
|---|---|---|
| `matcher` | 正则（对齐 Claude Code），用于匹配工具名/通知类型 | 所有 |
| `tools` | 工具名白名单（YAML 风格，`List<String>`） | `PRE_TOOL_USE` / `POST_TOOL_USE` / `POST_TOOL_USE_FAILURE` |
| `file_patterns` | glob 模式（`*.java`、`src/**/*.xml`） | 文件操作工具 |
| `agent_name` | Agent 名过滤 | `PRE_AGENT_SWITCH` / `POST_AGENT_SWITCH` / `SUBAGENT_STOP` |
| `error_pattern` | 错误消息正则 | `ON_ERROR` |

**HookContext stdin JSON**（源码 `HookContext.toStdinJson`，传给脚本的 JSON）：
```json
{
  "hook_event_name": "POST_TOOL_USE",
  "cwd": "/path/to/project",
  "session_id": "abc-123",
  "tool_name": "WriteFile",
  "tool_use_id": "call_xxx",
  "tool_input": { "path": "/x/y.java", "content": "..." },
  "tool_response": "...",
  "stop_hook_active": false
}
```

**Exit Code 决策**（源码 `HookExecutor`）：

| Exit Code | 含义 |
|---|---|
| `0` | 成功，允许操作继续 |
| `2` | **阻塞**操作（对齐 Claude Code，仅 `PRE_TOOL_USE` 有效） |
| 其他 | 非阻塞错误，只打 warn 日志不中断 |

### 4.4 注册方式

**加载目录**（源码 `HookLoader`）：

| 位置 | 优先级 | 说明 |
|---|---|---|
| `classpath:/hooks/*.yaml` | 低 | 内置 Hook（打进 JAR） |
| `~/.jimi/hooks/*.yaml` | 中 | 用户级 |
| `<project>/.jimi/hooks/*.yaml` | 高 | 项目级 |

**启动时即加载、热加载不支持**——新增/修改 Hook 后需重启 Jimi 会话。

### 4.5 踩坑提示

- ❌ **`exit 2` 只在 `PRE_TOOL_USE` 有效** → `POST_TOOL_USE` 里 exit 2 不能"撤销"已执行的工具
- ⚠️ **脚本里能访问哪些变量**：源码 `HookExecutor.buildVariableMap#442-478` 构建 `Map<String, String>`，然后由 `ScriptRunner.replaceVariables#149-155` 做 **Java 字符串替换**（`text.replace("${KEY}", value)`），再把处理后的脚本交给 `/bin/bash -c` 执行。所以：
  - **必须用 `${VAR}` 语法**，不是 `$VAR`（前者会被 Jimi 预替换，后者才会走 bash 环境变量解析）
  - 完整 13 个可替换变量：`${JIMI_WORK_DIR}` / `${HOME}` / `${TOOL_NAME}` / `${TOOL_RESULT}` / `${MODIFIED_FILE}`（首个文件）/ `${MODIFIED_FILES}`（空格分隔）/ `${AGENT_NAME}` / `${CURRENT_AGENT}` / `${PREVIOUS_AGENT}` / `${ERROR_MESSAGE}` / `${USER_INPUT}` / `${NOTIFICATION_MESSAGE}` / `${LAST_ASSISTANT_MESSAGE}`
  - **注意清单里没有 `${TOOL_INPUT}`**——工具入参（嵌套 Map）只通过 stdin JSON 传递，脚本需用 `jq` 解析：
  ```bash
  # 从 ${VAR} 字符串替换读（简单字段）- 在脚本交给 bash 之前就被 Jimi 替换成实际值
  echo "Modified: ${MODIFIED_FILE}"
  echo "Tool: ${TOOL_NAME}"
  
  # 从 stdin JSON 读（复杂嵌套字段，如 tool_input、affected_files）
  TOOL_INPUT=$(jq -r '.tool_input' </dev/stdin)
  ```
- ❌ **`async: true` 时依赖前一步输出** → 异步执行不保证顺序，如果后一个 Hook 要用前一个的输出，必须全部 `async: false`
- ❌ **`tools: []` vs 不写 `tools`** → 源码中 `tools` 默认 `new ArrayList<>()`，**空列表表示匹配所有工具**（不是"不匹配任何"）
- ❌ **`matcher` 正则语法错** → `validate()` 会在启动时抛 `IllegalArgumentException`，整个 Hook 被丢弃（只在日志里看到）
- ⚠️ **脚本里修改 Jimi 会话状态** → 脚本是独立进程，无法直接改 `Session` / `Context`。要改状态必须走 stdout JSON 输出 + Jimi 侧解析（详见 07 篇）
- ℹ️ **热加载支持**：`HookRegistry#reloadHooks()` 真实存在（源码 `core/hook/HookRegistry.java#337+`），但**需要主动触发**（无 `WatchService` 自动监听），目前仅 `/reload` 类内部调用路径会用到，CLI 面向用户未暴露命令 → 实操中修改 Hook 后仍建议重启会话

## §5 自定义 Slash 命令

### 5.1 两种实现路径

| 路径 | 契约 | 适用 |
|---|---|---|
| **YAML 声明式**（`/.jimi/commands/*.yaml`） | `CustomCommandSpec` + `ConfigurableCommandHandler` 自动适配 | 固定 prompt / 脚本 / 多步骤组合，无需 Java 逻辑 |
| **Java 编程式**（`@Component implements CommandHandler`） | `command/CommandHandler.java` 接口 | 需访问 Spring Bean、修改 Session、复杂分支逻辑 |

### 5.2 YAML 声明式命令

**最简示例：prompt 模板**（`~/.jimi/commands/review.yaml`）

```yaml
name: review
description: 让 AI 审查当前改动并给出建议
aliases:
  - rv
category: development
prompt: |
  请审查当前工作目录的 git 未提交改动：
  1. 使用 Bash 工具执行 `git diff --staged`
  2. 检查代码风格、潜在 bug、性能问题
  3. 给出改进建议

  用户额外要求：$ARGUMENTS
```

输入 `/review 重点看并发安全`，`$ARGUMENTS` 会被替换为 `重点看并发安全`，整段 prompt 作为新的 user message 发给 LLM。

**script 类型示例**：

```yaml
name: build
description: 执行 Maven 构建
aliases: [b]
execution:
  type: script
  script: mvn clean install -DskipTests
  timeout: 300                    # 秒
requireApproval: false            # 是否需要审批
```

**composite 类型**：将多个步骤串起来（先 script 再 agent），详见 07 篇 §15（`core/hook/ExecutionSpec.java` 支持 `script`/`agent`/`composite` 三种类型，由 `ConfigurableCommandHandler` 的 switch 分派）。

### 5.3 Java 编程式命令

参考 `command/handlers/VersionCommandHandler.java` 等内置命令的写法：

```java
package io.leavesfly.jimi.command.handlers;

import io.leavesfly.jimi.command.CommandContext;
import io.leavesfly.jimi.command.CommandHandler;
import org.springframework.stereotype.Component;

import java.util.List;

@Component
public class GitStatusCommandHandler implements CommandHandler {

    @Override
    public String getName() {
        return "gs";
    }

    @Override
    public String getDescription() {
        return "显示 git 状态";
    }

    @Override
    public List<String> getAliases() {
        return List.of("status");
    }

    @Override
    public int getPriority() {
        return 10;                    // 与 YAML 命令同名时，priority 高者胜
    }

    @Override
    public String getCategory() {
        return "git";
    }

    @Override
    public void execute(CommandContext context) throws Exception {
        // CommandContext 提供的 getter：getEngineClient/getTerminal/getLineReader
        // / getRawInput/getCommandName/getArgs/getOutputFormatter
        // 以及 getArgsAsString()/getArg(int)/getArgCount() 三个参数便捷方法
        Process p = new ProcessBuilder("git", "status", "--short")
                .redirectErrorStream(true)
                .start();
        String output = new String(p.getInputStream().readAllBytes());
        context.getOutputFormatter().println(output);   // ✅ 真实存在
    }

    @Override
    public boolean isAvailable(CommandContext context) {
        // 当前目录非 git 仓库时不可用
        // 注意：CommandContext 没有 getWorkDir()，要拿工作目录需通过
        // context.getEngineClient() 或 System.getProperty("user.dir")
        return java.nio.file.Files.exists(
                java.nio.file.Paths.get(System.getProperty("user.dir"), ".git"));
    }
}
```

**`CommandHandler` 接口契约**（源码 `command/CommandHandler.java`）：

| 方法 | 必需 | 默认 | 说明 |
|---|---|---|---|
| `getName()` | ✅ | — | 命令名（不含 `/` 前缀） |
| `getDescription()` | ✅ | — | 简短描述 |
| `execute(CommandContext)` | ✅ | — | 可抛 `Exception` |
| `getAliases()` | ⚪ | `[]` | 别名列表 |
| `getUsage()` | ⚪ | `/<name>` | 帮助文本 |
| `getPriority()` | ⚪ | `0` | 同名冲突时优先级高者胜出 |
| `getCategory()` | ⚪ | `"general"` | 帮助分组 |
| `isAvailable(ctx)` | ⚪ | `true` | 运行时动态可用性 |

### 5.4 踩坑提示

- ❌ **YAML `name` 与内置命令冲突** → 内置 Java 处理器优先级通常较高，YAML 版本会被忽略。真实内置命令清单（18 个，源码 `command/handlers/` 目录）：`new` / `help` / `init` / `wiki` / `clear` / `graph` / `index` / `reset` / `theme` / `tools` / `agents` / `config` / `memory` / `status` / `compact` / `history` / `version` / `commands`
- ❌ **YAML `execution` 和 `prompt` 都不填** → `validate()` 抛 "Either 'execution' or 'prompt' is required"，命令被丢弃
- ❌ **Java 命令写 `@Component` 但类不在 `io.leavesfly.jimi` 包下** → Spring 扫不到，启动时没错但 `/` 敲不出来该命令
- ❌ **`execute` 里做耗时操作不异步化** → CLI 线程会被阻塞，终端卡死；耗时操作应在 `execute` 内启子线程或用 `CompletableFuture.runAsync`
- ⚠️ **`$ARGUMENTS` 占位符只在 `prompt` 类型命令生效**（对齐 Claude Code） → `script` 命令要用参数得走 `parameters:` 定义 + `${paramName}` 占位符
- ⚠️ **`requireApproval: true` 的 YAML 命令** → 会走 Jimi 的 `Approval` 审批链路，yolo 模式（`--yolo`）下自动批准

## §6 LLM ChatProvider

### 6.1 适用场景

- 接入一个新的**国产大模型厂商**（如文心、豆包、讯飞等）
- 接入一个**私有部署**的模型服务（VLLM、SGLang、Ollama）
- 对现有 Provider 做定制（自定义鉴权、流量控制、A/B 测试）

**优先使用 OpenAICompatibleChatProvider**：如果目标服务已经支持 OpenAI Chat Completions API（`/v1/chat/completions`），**不需要写新 Provider**，只需在 `jimi.yaml` 加配置即可。实测下 DeepSeek、Qwen、Ollama 都走这条路。

**必须写新 Provider 的情况**：
- 服务使用完全不同的协议（如 Cursor 的 CLI 包装、MCP 直连）
- 需要特殊的流控/重试逻辑（如 Kimi 的 `RateLimiter`）
- 需要强制的请求改写（如 tool-calling 的本地模拟）

### 6.2 最小模板

参考 `KimiChatProvider` 和 `OpenAICompatibleChatProvider`。若要接入 OpenAI 兼容服务，**零代码改动**，只需在 `~/.jimi/jimi.yaml` 里加：

```yaml
providers:
  deepseek:
    type: DEEPSEEK
    base_url: https://api.deepseek.com
    api_key: ${DEEPSEEK_API_KEY}    # 或硬编码
  
models:
  deepseek-chat:
    provider: deepseek
    model: deepseek-chat
    max_context_size: 64000
```

然后 `jimi --model deepseek-chat` 或在 `default_model` 指定即可。

**若要写新 Provider**（以下代码结构直接对标 `KimiChatProvider.java` 的真实实现，仅改名为 `MyProvider`；所有方法均为可运行链路，非占位）：

```java
package io.leavesfly.jimi.llm.provider;

import com.fasterxml.jackson.databind.JsonNode;
import com.fasterxml.jackson.databind.ObjectMapper;
import com.fasterxml.jackson.databind.node.ArrayNode;
import com.fasterxml.jackson.databind.node.ObjectNode;
import io.leavesfly.jimi.config.info.LLMProviderConfig;
import io.leavesfly.jimi.llm.ChatCompletionChunk;
import io.leavesfly.jimi.llm.ChatCompletionResult;
import io.leavesfly.jimi.llm.ChatProvider;
import io.leavesfly.jimi.llm.message.FunctionCall;
import io.leavesfly.jimi.llm.message.Message;
import io.leavesfly.jimi.llm.message.ToolCall;
import lombok.extern.slf4j.Slf4j;
import org.springframework.http.MediaType;
import org.springframework.web.reactive.function.client.WebClient;
import reactor.core.publisher.Flux;
import reactor.core.publisher.Mono;

import java.util.ArrayList;
import java.util.List;

@Slf4j
public class MyProvider implements ChatProvider {

    private final String modelName;
    private final WebClient webClient;
    private final ObjectMapper objectMapper;

    public MyProvider(String model, LLMProviderConfig config, ObjectMapper mapper) {
        this.modelName = model;
        this.objectMapper = mapper;
        WebClient.Builder builder = WebClient.builder()
                .baseUrl(config.getBaseUrl())
                .defaultHeader("Authorization", "Bearer " + config.getApiKey())
                .defaultHeader("Content-Type", "application/json");
        if (config.getCustomHeaders() != null) {
            config.getCustomHeaders().forEach(builder::defaultHeader);
        }
        this.webClient = builder.build();
    }

    @Override
    public String getModelName() { return modelName; }

    @Override
    public Mono<ChatCompletionResult> generate(
            String systemPrompt, List<Message> history, List<Object> tools) {
        return Mono.defer(() -> {
            try {
                ObjectNode body = buildRequestBody(systemPrompt, history, tools, false);
                return webClient.post()
                        .uri("/chat/completions")
                        .contentType(MediaType.APPLICATION_JSON)
                        .bodyValue(body)
                        .retrieve()
                        .bodyToMono(JsonNode.class)
                        .map(this::parseResponse);
            } catch (Exception e) {
                log.error("Failed to generate chat completion", e);
                return Mono.error(new RuntimeException("generate failed", e));
            }
        });
    }

    @Override
    public Flux<ChatCompletionChunk> generateStream(
            String systemPrompt, List<Message> history, List<Object> tools) {
        return Flux.defer(() -> {
            try {
                ObjectNode body = buildRequestBody(systemPrompt, history, tools, true);
                return webClient.post()
                        .uri("/chat/completions")
                        .contentType(MediaType.APPLICATION_JSON)
                        .bodyValue(body)
                        .retrieve()
                        .bodyToFlux(String.class)
                        .filter(line -> line.contains("delta"))
                        // SSE 行以 "data: " 前缀打头，去掉后再做 JSON 解析
                        .map(line -> line.startsWith("data: ") ? line.substring(6) : line)
                        .filter(data -> !"[DONE]".equals(data))
                        .map(this::parseStreamChunk);
            } catch (Exception e) {
                log.error("Failed to generate streaming chat completion", e);
                return Flux.error(new RuntimeException("generateStream failed", e));
            }
        });
    }

    /** 构造 OpenAI 兼容的请求体；对接自家协议时在此替换字段名即可。 */
    private ObjectNode buildRequestBody(
            String systemPrompt, List<Message> history, List<Object> tools, boolean stream) {
        ObjectNode body = objectMapper.createObjectNode();
        body.put("model", modelName);
        body.put("stream", stream);

        ArrayNode messages = objectMapper.createArrayNode();
        if (systemPrompt != null && !systemPrompt.isEmpty()) {
            ObjectNode sys = objectMapper.createObjectNode();
            sys.put("role", "system");
            sys.put("content", systemPrompt);
            messages.add(sys);
        }
        for (Message msg : history) {
            // 真实工程里需要处理 content 为 List<ContentPart>、tool_calls、tool_call_id 等
            // 分支，完整实现见 KimiChatProvider#convertMessage
            messages.add(objectMapper.valueToTree(msg));
        }
        body.set("messages", messages);

        if (tools != null && !tools.isEmpty()) {
            body.set("tools", objectMapper.valueToTree(tools));
        }
        return body;
    }

    /** 解析非流式响应：choices[0].message → ChatCompletionResult */
    private ChatCompletionResult parseResponse(JsonNode response) {
        JsonNode messageNode = response.get("choices").get(0).get("message");
        String content = messageNode.hasNonNull("content") ? messageNode.get("content").asText() : null;
        String reasoning = messageNode.hasNonNull("reasoning_content")
                ? messageNode.get("reasoning_content").asText() : null;

        List<ToolCall> toolCalls = null;
        if (messageNode.has("tool_calls")) {
            toolCalls = new ArrayList<>();
            for (JsonNode tc : messageNode.get("tool_calls")) {
                toolCalls.add(ToolCall.builder()
                        .id(tc.get("id").asText())
                        .type(tc.has("type") ? tc.get("type").asText() : "function")
                        .function(FunctionCall.builder()
                                .name(tc.get("function").get("name").asText())
                                .arguments(tc.get("function").get("arguments").asText())
                                .build())
                        .build());
            }
        }
        Message msg = (toolCalls != null && !toolCalls.isEmpty())
                ? Message.assistant(content, reasoning, toolCalls)
                : Message.assistant(content, reasoning);

        ChatCompletionResult.Usage usage = null;
        if (response.has("usage")) {
            JsonNode u = response.get("usage");
            usage = ChatCompletionResult.Usage.builder()
                    .promptTokens(u.get("prompt_tokens").asInt())
                    .completionTokens(u.get("completion_tokens").asInt())
                    .totalTokens(u.get("total_tokens").asInt())
                    .build();
        }
        return ChatCompletionResult.builder().message(msg).usage(usage).build();
    }

    /** 解析 SSE 流式块：content 增量 / reasoning 增量 / tool_call 增量 / DONE */
    private ChatCompletionChunk parseStreamChunk(String data) {
        try {
            JsonNode chunk = objectMapper.readTree(data);
            JsonNode choice = chunk.get("choices").get(0);
            JsonNode delta = choice.get("delta");

            if (choice.hasNonNull("finish_reason")) {
                return ChatCompletionChunk.builder()
                        .type(ChatCompletionChunk.ChunkType.DONE).build();
            }
            if (delta.hasNonNull("content")) {
                return ChatCompletionChunk.builder()
                        .type(ChatCompletionChunk.ChunkType.CONTENT)
                        .contentDelta(delta.get("content").asText())
                        .build();
            }
            if (delta.has("tool_calls")) {
                JsonNode tc = delta.get("tool_calls").get(0);
                return ChatCompletionChunk.builder()
                        .type(ChatCompletionChunk.ChunkType.TOOL_CALL)
                        .toolCallId(tc.has("id") ? tc.get("id").asText() : null)
                        .functionName(tc.path("function").path("name").asText(null))
                        .argumentsDelta(tc.path("function").path("arguments").asText(null))
                        .build();
            }
            return ChatCompletionChunk.builder()
                    .type(ChatCompletionChunk.ChunkType.CONTENT).contentDelta("").build();
        } catch (Exception e) {
            log.error("Failed to parse stream chunk: {}", data, e);
            return ChatCompletionChunk.builder()
                    .type(ChatCompletionChunk.ChunkType.CONTENT).contentDelta("").build();
        }
    }
}
```

> **说明**：以上代码按 `KimiChatProvider.java`（381 行）的真实链路精简抽取，保留 **请求构造 / 响应解析 / 流式解析 / 工具调用解析** 四条核心路径；完整实现（包括 `ContentPart` 分支、无效 `tool_calls` 过滤、`RateLimiter`、`reasoning_content` 增量、`usage` 统计）请直接阅读 `KimiChatProvider.java`。

### 6.3 契约速览

**`ChatProvider` 接口**（源码 `llm/ChatProvider.java`，**只有 3 个方法，全部必需**）：

| 方法 | 返回 | 说明 |
|---|---|---|
| `String getModelName()` | — | 返回模型标识（如 `"moonshot-v1-32k"`） |
| `Mono<ChatCompletionResult> generate(system, history, tools)` | 单个完整响应 | **非流式**，适合内部工具调用（如子 Agent） |
| `Flux<ChatCompletionChunk> generateStream(system, history, tools)` | SSE 流 | **流式**，Jimi 交互式 CLI 默认走这个 |

⚠️ **`generateStream` 不能省略**——Jimi 的 CLI 交互式会话默认走流式。只实现 `generate` 会导致启动时 NPE 或流式输出缺失。

**`LLMProviderConfig.ProviderType`** 枚举控制 `LLMFactory.createChatProvider` 的分派。源码 `config/info/LLMProviderConfig.java#79-99` 共 **9 个枚举值**：`KIMI` / `DEEPSEEK` / `QWEN` / `OLLAMA` / `OPENAI` / `CLAUDE` / `CURSOR` / `GLM` / `MINIMAX`。分派规则（源码 `LLMFactory.createChatProvider#169-184`）：
- `KIMI` → `KimiChatProvider`（带 `RateLimiter`）
- `CURSOR` → `CursorChatProvider`（cursor-agent CLI 包装）
- **其他全部 7 个**（`DEEPSEEK` / `QWEN` / `OLLAMA` / `OPENAI` / `CLAUDE` / `GLM` / `MINIMAX`） → 走 `OpenAICompatibleChatProvider`，`providerName` 由 `formatProviderLabel` 按大小写规则生成（如 `MINIMAX` → `MiniMax`）。

### 6.4 注册方式

**方式 1：加入现有 ProviderType**

1. 在 `config/info/LLMProviderConfig.java#ProviderType` 枚举里加 `MYPROVIDER`
2. 在 `LLMFactory.createChatProvider` 的 switch 里加 `case MYPROVIDER: return new MyProvider(...)`
3. 在 `jimi.yaml` 配置 `type: MYPROVIDER`

**方式 2：绕过枚举直接注册**

目前 Jimi 的 `LLMFactory` 是 `@Service` 单例，没有提供 `registerProvider` 运行时注册 API——**必须改源码**。社区贡献时建议走方式 1。

**API Key 解析顺序**（源码 `LLMFactory.resolveApiKey`）：
1. 环境变量 `{TYPE}_API_KEY`（如 `DEEPSEEK_API_KEY`）**优先**
2. 配置文件 `providers.xxx.api_key` 次之
3. 配置文件值为 `"xxxx"` 或空 → 视为未配置

**Fail-fast 校验**：目前仅 `QWEN` 类型在 key 缺失时启动抛 `ConfigException`（源码 `LLMFactory.createLLM` 的 `if ((apiKey == null || apiKey.isEmpty()) && providerConfig.getType().equals(LLMProviderConfig.ProviderType.QWEN))` 分支），其他 8 个类型仅打 debug 日志——这是一个已知的不对称行为（05 篇 §3 `LLMFactory` 的 API Key 解析链路有详细说明）。

### 6.5 踩坑提示

- ❌ **忘记实现 `generateStream`** → CLI 交互式输出会卡住或报 "generateStream not implemented"
- ❌ **在 `generate` 内用 `.block()`** → 会阻塞 Reactor 线程池；正确做法是返回 `webClient.post().retrieve().bodyToMono(...)` 响应式链
- ❌ **响应体字段与 `ChatCompletionResult` 不匹配** → 静默返回空消息。必须严格按 OpenAI 格式（`choices[0].message.content`、`choices[0].message.tool_calls`）映射
- ❌ **流式 SSE 解析未去掉 `data: ` 前缀** → `objectMapper.readValue` 抛 JSON 解析错误，流式中断
- ⚠️ **Tool Calling 格式** → 不同厂商对 `tools` 参数的接收格式不同（OpenAI 要 `tools: [{type:function, function: {name, parameters}}]`，有些厂商只要 `functions: [...]`）；参考 `OpenAICompatibleChatProvider#buildRequestBody` 的格式处理
- ⚠️ **LLM 实例 Caffeine 缓存** → `LLMFactory.llmCache` 按模型名缓存 30 分钟，修改 Provider 代码后需重启或调 `clearCache()`

## §7 ToolProvider SPI

### 7.1 适用场景

- 一次注入**一组工具**（如把整个"代码图谱查询套件"作为一个插件注册）
- 需要按 **AgentSpec** 动态决定工具集（某些工具只给特定 Agent 用）
- 工具有**共享的生命周期**（同源、同认证、同清理逻辑，如 MCP 的 `activeClients`）
- 工具来源于**外部可配置源**（插件市场、MCP Server、远程注册表）

**不适合的场景**：
- 只加 1-2 个独立工具 → 直接走 §2 `@Component` + `ToolRegistryFactory` 注册
- 工具完全静态（编译期就知道） → §2 更直接

### 7.2 最小模板

参考 `MCPToolProvider.java`（真实的 SPI 实现）：

```java
package io.leavesfly.jimi.tool.provider;

import io.leavesfly.jimi.core.agent.AgentSpec;
import io.leavesfly.jimi.core.engine.JimiRuntime;
import io.leavesfly.jimi.tool.Tool;
import io.leavesfly.jimi.tool.ToolProvider;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Component;

import java.util.ArrayList;
import java.util.List;

@Slf4j
@Component    // ⚠️ 必须是 Spring Bean，否则 ToolRegistryFactory 扫不到
public class MyPluginToolProvider implements ToolProvider {

    @Override
    public boolean supports(AgentSpec agentSpec, JimiRuntime jimiRuntime) {
        // 仅为名为 "coding" 的 Agent 提供工具
        return "coding".equals(agentSpec.getName());
    }

    @Override
    public List<Tool<?>> createTools(AgentSpec agentSpec, JimiRuntime jimiRuntime) {
        List<Tool<?>> tools = new ArrayList<>();
        try {
            // 从插件市场/远程注册表拉取 → 实例化 → 加入列表
            tools.add(new MyRemoteTool("https://plugin.example.com/xxx"));
            tools.add(new MyLocalTool(jimiRuntime.getSession()));
            log.info("Loaded {} tools from MyPlugin", tools.size());
        } catch (Exception e) {
            log.error("Failed to load MyPlugin tools: {}", e.getMessage());
            // 关键：异常不要抛，返回空列表，避免拖垮整个 ToolRegistry 构建
        }
        return tools;
    }

    @Override
    public int getOrder() {
        return 80;    // 在 MCPToolProvider(60) 之后加载
    }

    @Override
    public String getName() {
        return "MyPluginToolProvider";
    }
}
```

### 7.3 契约速览

**`ToolProvider` 接口**（源码 `tool/ToolProvider.java`）：

| 方法 | 必需 | 默认 | 说明 |
|---|---|---|---|
| `boolean supports(agentSpec, runtime)` | ✅ | — | 返回 `false` 则整个 Provider 被跳过 |
| `List<Tool<?>> createTools(agentSpec, runtime)` | ✅ | — | 返回工具列表，空列表可接受 |
| `int getOrder()` | ⚪ | `100` | 数值越小越先加载（冲突时后者覆盖前者） |
| `String getName()` | ⚪ | 类名 | 仅用于日志 |

**分发流程**（源码 `ToolRegistryFactory.applyToolProviders`）：

```
1. Spring 自动注入所有实现 ToolProvider 的 @Component，按 @Autowired List<ToolProvider> 传入
2. 对 MCPToolProvider、MetaToolProvider 做特殊前置注入（setMcpConfigFiles / setToolRegistry）
3. stream().sorted(Comparator.comparingInt(ToolProvider::getOrder))
4. .filter(p -> p.supports(agentSpec, runtime))
5. .forEach(p -> registry.register(每个 tool))
```

### 7.4 `getOrder` 顺序表

真实仓库里的两个内置 Provider（已核实源码）：

| Provider | `getOrder()` | 源码位置 | 说明 |
|---|---|---|---|
| `MCPToolProvider` | `60` | `MCPToolProvider.java#83` | **早加载**：先注册 MCP 工具，后续自定义 ToolProvider 可基于 MCP 工具做组合 |
| `MetaToolProvider` | `200` | `MetaToolProvider.java#92` | **晚加载**：MetaTool 需要"看到"所有已注册的工具才能做元操作（JavaDoc 明写"确保其他工具已经注册"） |

**约定推荐值**（基于内置 Provider `MCPToolProvider=60` 和 `MetaToolProvider=200` 的取值）：

| 区间 | 用途 |
|---|---|
| 0-50 | 核心系统 Provider（保留） |
| 50-80 | 外部源 Provider（如 `MCPToolProvider=60`） |
| 80-150 | 自定义插件 Provider（推荐写在这里） |
| 150-250 | 需要依赖其他工具已注册的元工具 Provider（如 `MetaToolProvider=200`） |

### 7.5 踩坑提示

- ❌ **忘记 `@Component`** → `ToolRegistryFactory` 的 `@Autowired List<ToolProvider>` 扫不到，Provider 形同不存在
- ❌ **`createTools` 里抛异常** → 整个 `ToolRegistry` 构建失败，Jimi 启动报错退出。必须 try-catch 后返回空列表
- ❌ **返回的 `Tool` 实例有状态且被多次调用** → 不同会话可能共享，需确保工具内部线程安全（或每次 `createTools` 都 new 新实例）
- ❌ **依赖的 Spring Bean 用字段注入但没写 `@Autowired`** → 构造器注入推荐；若用字段注入忘记 `@Autowired`，Provider 能加载但 Bean 为 null，`createTools` 时 NPE
- ⚠️ **同一工具名被多个 Provider 注册** → 后注册的覆盖前面的（`ToolRegistry.register` 行为），`getOrder` 更大的 Provider 工具会"赢"
- ⚠️ **`supports` 方法有副作用** → `supports` 可能被调多次（多个 Agent 创建时），不要在里面做缓存写入或计数等副作用操作

## §9 常见踩坑清单

按扩展点聚合的**高频踩坑**（跳转到各章节可看详细解释）：

### Tool（§2）
1. 忘记 `@Scope("prototype")` → 工具被 Spring 复用为单例，多会话状态污染
2. `execute` 里直接阻塞 → 卡住 Reactor 线程池，必须 `subscribeOn(boundedElastic)`
3. 参数类非 `public static class` + Lombok → 反射生成的 schema 退化
4. 写操作没设 `isConcurrentSafe=false` → 并发调用导致文件冲突

### Skill（§3）
5. YAML Front Matter 的 `---` 两端带空格 → 正则匹配失败整个文件解析失败
6. YAML 用 Tab 缩进 → 解析失败被静默跳过
7. `name` 含空格/中文 → SkillsTool 激活失败
8. `content` 过长 → 激活时 token 爆表

### Hook（§4）
9. `exit 2` 只对 `PRE_TOOL_USE` 生效 → 用错 HookType 导致"拦截"失效
10. 脚本里必须用 `${VAR}` 不是 `$VAR`（是 Jimi 预先字符串替换，不是 bash 环境变量）；且 `${TOOL_INPUT}` 不存在，要嵌套字段只能 `jq` 解析 stdin JSON
11. `tools: []` 不等于"不匹配任何" → 空列表表示**匹配所有工具**
12. `HookRegistry.reloadHooks()` 存在但 CLI 未暴露触发命令 → 实操中修改 Hook 后需重启会话

### Slash 命令（§5）
13. YAML 命令 `name` 与内置冲突 → 被内置 Java handler 覆盖
14. 同时不填 `execution` 和 `prompt` → validate 抛错命令被丢弃
15. `$ARGUMENTS` 只在 prompt 类型生效 → script 类型要用 `parameters:` + `${name}`

### LLM ChatProvider（§6）
16. 忘记实现 `generateStream` → CLI 流式输出卡住
17. 用 `.block()` 同步调用 → 阻塞 Reactor
18. SSE 流式解析未去 `data: ` 前缀 → JSON 解析错误流中断
19. 只有 `QWEN` 启动校验 API Key → 其他 Provider 缺 key 不会 fail-fast

### ToolProvider SPI（§7）
20. 忘记 `@Component` → `@Autowired List<ToolProvider>` 扫不到
21. `createTools` 抛异常未捕获 → 拖垮整个 ToolRegistry 构建
22. `supports` 有副作用 → 多次调用时行为异常

### MCP（见 11 篇）
23. `--mcp-config-file "a,b"` 不生效 → 必须重复传参
24. `HttpJsonRpcClient` 构造器不做网络探测 → URL 不可达的 client 仍进 `activeClients`
25. MCP 协议版本硬编码 `2024-11-05` → 服务端要求更高版本会失败

## §10 小结

本篇从**动手操作**视角覆盖了 Jimi 的 6 大扩展点，回答了"**给 Jimi 加 X，该怎么做**"的核心问题。

### 10.1 扩展点选型总览

```
需求维度                        推荐扩展点
──────────────────────────────  ──────────────────
增强 LLM 调用能力（单个工具）    §2 自定义 Tool
增强 LLM 调用能力（一组工具）    §7 ToolProvider SPI
增强 LLM 调用能力（外部协议）    配置 MCP Server（见 11 篇）
改变 LLM 行为（静态知识）        §3 Skill
改变 LLM 行为（动作触发）        §4 Hook
改变 CLI 交互（模板指令）        §5.2 YAML 命令
改变 CLI 交互（Java 逻辑）       §5.3 Java 命令
接入新大模型（OpenAI 兼容）      §6 改 jimi.yaml（零代码）
接入新大模型（自定义协议）       §6 新 ChatProvider 类
```

### 10.2 设计哲学观察

回顾 6 大扩展点，Jimi 的扩展性设计呈现出几个一致的模式：

**1. 双通路并存**：几乎每个扩展点都提供**配置式**和**代码式**两条路（Skill/Hook/YAML 命令 vs Tool/ToolProvider/Java 命令），让不同技能等级的贡献者都能参与。

**2. 三层加载**：Skill、Hook、Command 统一采用 `classpath → ~/.jimi/ → project/.jimi/` 的三层加载模式，项目级最高优先级，便于跨项目复用和项目定制。

**3. Spring SPI 贯穿**：`ToolProvider`、`CommandHandler`、`ChatProvider` 都通过 Spring `@Component` + `@Autowired List<T>` 的模式做松散耦合，新增扩展**不需要改中心化注册表**。

**4. Reactor 全链路响应式**：工具、LLM、Hook 执行统一走 `Mono`/`Flux`，配合 `@Scope("prototype")` 保证会话隔离。

**5. Claude Code 对齐 + Jimi 扩展**：Hook 类型、命令格式、MCP 协议都明确对标 Claude Code 规范，在此基础上做本地扩展（Jimi 扩展的 3 个 Hook 类型、Teams 扩展的 3 个类型）。
